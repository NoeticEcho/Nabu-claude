# /nabu-library

Управляй **библиотекой справочных знаний** Nabu — это книги, документы, тексты и страницы,
которые НЕ про пользователя (напр. учебник по КПТ, книга по UI/UX, база проектов законов РФ).
Библиотека — отдельное хранилище от личной памяти пользователя: агенты и Совет ищут в ней как
в СВОЁМ справочнике, скоупясь по домену.

> **Библиотека ≠ личная память.** Личная память (7 типов) — факты о пользователе (`kind=personal`).
> Библиотека — внешние знания (`kind=library`, `visibility=default`). Никогда не путать: контент
> библиотеки не является фактом о пользователе, а личные заметки не кладутся в библиотеку.
> `vault` в библиотеку не принимается (сервер отклонит).

## Подкоманды

### `add <путь-или-URL> --domain <домен> [--title <название>]`
Проиндексировать локальный **файл/папку** ИЛИ **URL** как справочное знание домена. Тяжёлое
(чтение/чанкинг/эмбеддинги, Ollama 768) выполняется ЛОКАЛЬНО на сервере.

1. **Файл/папка** → `nabu-pipeline.add_library_source({ path, domain, title? })`. Для папки —
   каждый файл индексируется отдельно (лимиты как у `/nabu-index`: ≤500 файлов, ≤2МБ).
2. **URL** → `nabu-pipeline.add_library_source({ url, domain, title? })`. Сервер загружает
   страницу локально, извлекает текст и индексирует. Внешняя загрузка — объяви действие.
3. **Домен обязателен** (`psychology`, `uiux`, `law`, …) — это scope, по которому агенты потом
   ищут. Осмысленный `title` помогает в `list`/цитировании.
4. `kind=library`, `visibility=default` проставляются сервером. Веди счётчик добавленных
   документов/чанков и покажи итог.

### `list [--domain <домен>]`
`nabu-pipeline.list_library_sources({ domain? })` — что уже в библиотеке (источники, домены,
кол-во чанков). Без `--domain` — все домены.

### `search <запрос> [--domain <домен>]`
`nabu-pipeline.search_knowledge({ query, kind: "library", domain? })` — семантический поиск
ТОЛЬКО по библиотеке (не по личной памяти). `--domain` сужает до одного справочника. Верни
top-k с источником и близостью.

## Дистилляция навыков и порождение агента

После наполнения домена библиотеки можно **извлечь из него навык** и, если домен явно требует
выделенного эксперта, **породить доменного агента**:

- Диспатчь субагента **`library-curator`** с `{ domain }`. Он читает знания домена
  (`search_knowledge kind=library domain=X`), синтезирует конкретный actionable-навык (методы/
  фреймворки из источников — НЕ дословное копирование) и предлагает создать skill.
- Если домен тянет на отдельного эксперта (напр. `law` → «законотворец»), curator предлагает
  нового доменного агента через **`agent-creator`**, ссылающегося на домен библиотеки + навык.
- Артефакты (текст навыка, дефиниция агента, запись реестра) curator ВОЗВРАЩАЕТ адъютанту;
  файлы после ревью `critic` пишет адъютант (curator сам файлы не пишет).

## Границы и приватность
- Библиотека — только НЕличный контент. Личные данные пользователя → личная память
  (`/nabu-index`, `/nabu-recall`), не сюда.
- `vault` в библиотеку не принимается. URL-загрузка — внешнее действие, объяви его.
- Навык — синтез компетенции, не перепечатка источника; границы компетенции (`SAFETY.md`)
  сохраняются: агент из домена даёт информацию/структуру, не профессиональный совет.
