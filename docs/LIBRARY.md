# Библиотека знаний (reference)

Библиотека — это **справочное знание Совета**: книги, документы, тексты и веб-страницы, которые
НЕ про пользователя. Учебник по КПТ, книга по UI/UX, база проектов законов РФ — всё это внешние
знания, из которых агенты и Совет черпают компетенцию. Библиотека отделена от личной памяти
пользователя и живёт как собственный справочник агентов, скоупясь по домену.

## Библиотека ≠ личная память

Nabu держит два РАЗНЫХ хранилища, и путать их нельзя:

| | Личная память | Библиотека |
|---|---|---|
| Про кого | о пользователе (факты, эпизоды, цели) | внешние знания, НЕ про пользователя |
| `kind` | `personal` | `library` |
| Видимость | `private`/`vault`/`default` (по умолч. `private`) | `default` (не бывает личной) |
| Источник | диалог, `/nabu-index`, домен | книги/документы/URL через `/nabu-library` |
| Кто читает | адъютант при ответах о пользователе | агенты/Совет как свой справочник |
| Команды | `/nabu-recall`, `/nabu-index` | `/nabu-library` |

**Инвариант (жёстко):** контент библиотеки НИКОГДА не трактуется как факт о пользователе, а личные
данные пользователя не кладутся в библиотеку. Личный `recall` и поиск по библиотеке — разные store'ы
(`search_knowledge` фильтруется по `kind`). `vault` в библиотеку не принимается (сервер отклоняет).

## Как добавить источники

Команда — `/nabu-library add <путь-или-URL> --domain <домен> [--title <название>]`. CLI —
`nabu library add <path-or-url> --domain <d> [--title <t>]`.

- **Файл/папка** → `nabu-pipeline.add_library_source({ path, domain, title? })`. Папка индексируется
  по файлам (лимиты как у `/nabu-index`: ≤500 файлов, ≤2МБ на файл).
- **URL** → `nabu-pipeline.add_library_source({ url, domain, title? })`. Страница загружается и
  извлекается ЛОКАЛЬНО; это внешнее действие — адъютант его объявляет.
- Тяжёлое (чтение/чанкинг/эмбеддинги, Ollama 768, task-префиксы) выполняется на сервере — контент
  не гоняется через Claude. Сервер проставляет `kind=library`, `visibility=default`.

## Домены

Каждый источник тегируется **доменом** (`psychology`, `uiux`, `law`, …) — это scope, по которому
агенты потом ищут свой справочник. Домен обязателен при добавлении. Осмысленный `title` помогает в
`list` и цитировании. Список того, что уже есть, — `/nabu-library list [--domain <d>]`
(`nabu-pipeline.list_library_sources`).

## Как агенты используют библиотеку

Любой агент или министр Совета ищет справочные знания через
`nabu-pipeline.search_knowledge({ query, kind: "library", domain? })` — семантический поиск ТОЛЬКО
по библиотеке, суженный до нужного домена. Так министр `mind` опирается на домен `psychology`, а
специалист по интерфейсам — на `uiux`, не смешивая это с фактами о пользователе. CLI-эквивалент —
`nabu library search <query> [--domain <d>]`. Граф TypeDB связывает концепты библиотеки так же, как
и остальную базу знаний.

## Дистилляция навыка и порождение агента

Наполненный домен библиотеки можно превратить в применимую компетенцию — субагентом
**`library-curator`** (`agents/library-curator.md`):

1. Curator читает домен (`search_knowledge kind=library domain=X`, `list_library_sources`).
2. **Дистиллирует навык** — синтезирует методы/фреймворки/чек-листы источников в конкретный
   actionable skill (по форме `skills/nabu-*`), НЕ дословно копируя. Навык ссылается на домен-источник.
3. Если домен явно тянет на выделенного эксперта — curator предлагает **нового доменного агента**
   через `agent-creator`, ссылающегося на домен библиотеки + навык.
4. Артефакты возвращаются адъютанту; после ревью `critic` файлы и запись реестра пишет адъютант
   (curator сам файлы не пишет — `disallowedTools: Write, Edit, Bash`).

**Примеры:**
- Домен `psychology` (книга по КПТ) → навык «структура КПТ-разбора мысли» для министра `mind` (в
  границах SAFETY.md: информация и структура, не терапия/диагноз, направление к специалисту).
- Домен `law` (база проектов законов РФ) → агент **«законотворец»**, опирающийся на этот домен: даёт
  информацию и структуру по законотворчеству, НЕ юридическое заключение; направляет к юристу.

## Приватность и границы

- Библиотека — только НЕличный контент; `visibility=default`, `kind=library`. Личные данные →
  личная память, не сюда. `vault` в библиотеку не принимается.
- URL-загрузка — внешнее действие; приватность веб-запросов (не отправлять личное/`vault`) —
  как в общем правиле веб-доступа.
- Навык — синтез компетенции, не перепечатка источника. Границы компетенции (`SAFETY.md`)
  сохраняются: агент из домена библиотеки даёт информацию и структуру, не профессиональный совет,
  и направляет к лицензированному специалисту в high-risk доменах (здоровье/финансы/право/психика).
- Не выдумывать: если в домене нет данных под запрос — честно сказать, что справочника нет.
