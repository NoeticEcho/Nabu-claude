# 1. Введение

## 1.1. Назначение документа

Документ определяет требования к программной системе Nabu — мультиагентной среде для личной работы со знаниями, психологической работы, ведения проектов и саморазвития. Документ предназначен для команды разработки (включая автономных AI-coding агентов, работающих через оркестратор типа Aperant), для проектировщика архитектуры, для команды тестирования и для владельца продукта.

Структура документа следует ISO/IEC/IEEE 29148:2018, раздел 9.6 (Software requirements specification).

## 1.2. Область действия

Программная система называется Nabu. Включает в себя:

- Серверную часть на NestJS с интегрированной мультиагентной средой Mastra.ai.
- Клиентскую часть на Next.js (веб) и Tauri 2.x (десктоп и мобильные платформы).
- Backend-инфраструктуру на Supabase (self-hosted): Postgres, GoTrue Auth, Realtime, Storage, Edge Functions, Queues (pgmq), pg_cron.
- Графовую базу знаний TypeDB 3.0.
- Объектное хранилище S3 (MinIO для self-host).
- Опциональный локальный LLM через Ollama на десктопе.
Документ не описывает: маркетинговое позиционирование, ценообразование, юридические договоры. Эти вопросы вынесены в отдельные документы.

## 1.3. Перспектива продукта

Nabu — самостоятельный продукт. Не является надстройкой, плагином или расширением какого-либо существующего инструмента. Может импортировать данные из Obsidian, Logseq, Roam Research, Notion-export и обычных Markdown-корпусов. Может экспортировать всё содержимое в стандартный Markdown с YAML frontmatter.

## 1.4. Функции продукта (краткое перечисление)

- Захват и хранение разнотипных записей пользователя (заметки, дневники, идеи, задачи, ссылки, медиа).
- Автоматическая обработка ввода через мультиагентный конвейер (классификация, извлечение сущностей, связывание, обогащение).
- Ведение полиморфного графа знаний и адаптивной таксономии.
- Поддержка специализированных журналов (саморазвитие, психотерапия — КПТ, гештальт, DBT, ACT, IFS, и другие).
- Управление задачами, привычками, проектами, целями.
- Отслеживание метрик пользователя (финансы, физиология, ментальное состояние).
- Исследовательский режим (источники, цитаты, литературные обзоры).
- Генерация вторичных и третичных документов (спецификации, паспорта, дайджесты, манифесты).
- RPG-слой геймификации (статы, классы, квесты, награды).
- Многоплатформенная работа (веб, десктоп, мобильные) с локальным приоритетом и синхронизацией.

## 1.5. Характеристики пользователей

Основные пользовательские роли:

- Личный пользователь (single-user): один человек ведёт собственное пространство. Большинство сценариев.
- Терапевт-самостоятельный (subset личного): пользователь ведёт психотерапевтическую работу самостоятельно или параллельно с очной терапией; требует особой приватности и грамотной поддержки протоколов.
- Исследователь: пользователь ведёт исследовательский проект (научный, журналистский, продуктовый), активно работает с источниками и цитатами.
- Администратор self-host инстанса: технический пользователь, разворачивает и поддерживает экземпляр Nabu (часто совпадает с личным пользователем).
Предполагаемый уровень технической подготовки: от умеренного (умеет работать с Markdown и облачными приложениями) до высокого (для self-host).

## 1.6. Допущения и ограничения

- Платформенное ограничение: Tauri mobile — стабилен, но не «first-class citizen»; не все desktop-плагины портированы. Мобильная функциональность урезана относительно десктопа.
- LLM-ограничение: качество мультиагентного конвейера зависит от качества используемых моделей; eval-показатели задаются по состоянию Claude 4.6/4.7 и Qwen 2.5.
- Регуляторное ограничение: Nabu — не медицинский продукт. Психотерапевтические агенты — это поддержка собственной практики пользователя, а не клиническая интервенция. Маркируется явно во всех сценариях.
- Self-host ограничение: для полной офлайн-работы с локальным LLM требуется десктоп с не менее 16 ГБ ОЗУ и желательно GPU 8 ГБ+.

# 3. Конкретные требования

Соглашения по нотации требований:

- ID функциональных требований: FR-AABB, где AA — группа, BB — порядковый номер. Группы: 10 Аутентификация, 20 Заметки, 30 Конвейер, 40 Журналы, 50 Жизнь, 60 Метрики, 70 Исследования, 80 Платформы, 90 Геймификация.
- ID нефункциональных: NFR-AABB. Группы по ISO 25010: 10 Performance, 20 Reliability, 30 Security, 40 Usability, 50 Compatibility, 60 Maintainability, 70 Portability, 80 Functional suitability, 90 Privacy.
- Приоритет (Pri): M — Must (обязательно), S — Should (следует), C — Could (можно).
- Этап (Phase): Ph0…Ph7 по «Функциональной матрице».
- Метод верификации (V): T — Test, A — Analysis, I — Inspection, D — Demonstration.

## 3.1. Внешние интерфейсы

### 3.1.1. Пользовательский интерфейс

- Web: Next.js (App Router), адаптивная разметка, темы (light/dark/system), Tailwind.
- Desktop: Tauri 2.x WebView с тем же фронтенд-кодом + нативные слои (file watcher, hotkeys, native menus, notifications).
- Mobile: Tauri 2.x с адаптивной разметкой + нативные плагины (share-extension, push, биометрия).

### 3.1.2. Программные интерфейсы (API)

- REST/HTTP — NestJS API (OpenAPI 3.1 спецификация генерируется автоматически).
- Server-Sent Events / streaming — для агентских ответов через Mastra.
- WebSocket — Supabase Realtime для уведомлений и presence.
- MCP (Model Context Protocol) — для подключения внешних коннекторов.

### 3.1.3. Аппаратные интерфейсы

- Минимум для сервера self-host: 4 CPU vCore, 8 ГБ ОЗУ, 100 ГБ SSD.
- Рекомендуемое для сервера с локальным LLM: добавьте GPU NVIDIA 12+ ГБ VRAM.
- Минимум для десктоп-клиента: 4 ГБ ОЗУ. С локальным LLM: 16 ГБ ОЗУ, GPU 8+ ГБ VRAM.
- Минимум для мобильного клиента: iOS 15+ / Android 10+.

### 3.1.4. Коммуникационные интерфейсы

- HTTPS / TLS 1.3 для всех клиент-серверных коммуникаций.
- WSS для realtime.
- S3-протокол для объектного хранилища (MinIO API-совместимый).

# 4. Верификация требований

Каждое функциональное и нефункциональное требование имеет указанный метод верификации (T/A/I/D).

- T (Test) — автоматический тест: unit, integration, e2e. Включает eval-тесты для агентов.
- A (Analysis) — анализ кода, конфигурации, архитектурных артефактов. Включает security audit.
- I (Inspection) — ручная инспекция документации, кода, политик.
- D (Demonstration) — ручная демонстрация работы функции в собранном продукте.
Для каждой пары (требование, метод) ведётся отдельная запись в матрице трассируемости (см. Приложение А).

## 4.1. Категории тестов

- Unit-тесты — для каждой бизнес-функции, для каждой утилиты. Цель: ≥ 70% покрытия.
- Integration-тесты — для каждого API-эндпоинта, для каждой пары агент-инструмент.
- E2E-тесты — для ключевых сценариев пользователя (регистрация → импорт → создание заметки → агентская обработка → результат).
- Eval-тесты для агентов — отдельный класс, измеряющий качество выходов LLM-агента на корпусе с золотым стандартом. Метрики: accuracy, precision, recall, f1, или специфические для агента.
- Performance-тесты — нагрузочные сценарии (1, 10, 100, 1000 пользователей одновременно).
- Security-тесты — SAST (CodeQL/Semgrep), DAST (OWASP ZAP), dependency scan (Trivy, npm audit).
