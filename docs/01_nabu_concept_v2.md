# 0. Резюме изменений относительно v1

Версия 1 концепции описывала Nabu как веб-приложение на Next.js с серверным мультиагентным конвейером. Версия 2 расширяет архитектуру в четырёх направлениях, оставляя ядро (CopilotKit, Mastra.ai, TypeDB, Markdown в S3, RPG-слой) без изменений.

1. Многоплатформенность. Кроме веб-клиента появляются десктопный и мобильный клиенты на Tauri 2.x. Единый фронтенд-код, разные сборки. Это даёт нативную интеграцию с ОС, локальное хранение Markdown-файлов и опциональный локальный LLM на десктопе.
1. Backend на NestJS. Серверная часть строится не как Next.js API-routes, а как самостоятельный NestJS-приложение, в которое Mastra.ai встроена через официальный адаптер @mastra/nestjs. Это даёт DI, модульность, явные контроллеры, готовый rate limiting, MCP transport и предсказуемую структуру кода под автономную AI-разработку.
1. Supabase как полный backend-слой (с приоритетом self-hosted). Auth, Postgres, pgvector, Realtime, Storage, Edge Functions, Queues (pgmq), pg_cron — все компоненты доступны в self-hosted сборке через docker-compose. Это исключает vendor lock и позволяет работать на собственном железе.
1. Local-first гибридная модель. Часть данных пользователя живёт локально на устройстве (.md-файлы, локальный SQLite-кэш, опциональный локальный LLM); сервер — источник истины и координатор синхронизации. Server-side остаётся: реляционная БД, граф знаний, ИИ-агенты, тяжёлые модели.

# 1. Целевые платформы и стратегия поставки

Nabu — единое продуктовое пространство, доступное с трёх категорий устройств. Архитектура поставки сводится к одному общему UI-коду и трём дистрибутивам.

| **Платформа** | **Сборка** | **Особенности** |
| --- | --- | --- |
| Web (десктопный браузер) | Next.js, SSR/RSC, обычный деплой | Эталон функциональности. Хранение — только удалённое (S3). Локальный LLM недоступен. |
| Desktop (Windows/macOS/Linux) | Tauri 2.x + тот же Next.js-фронтенд (через статическую сборку SPA или встроенный HTTP) | Локальное хранение .md, локальный SQLite-кэш, опциональный Ollama для локального LLM. Глубокие интеграции (file watcher, системные уведомления, hotkeys, scheduling). |
| Mobile (iOS, Android) | Tauri 2.x mobile + тот же фронтенд (адаптивная разметка) | Локальное хранение .md (с ограничениями), кэш SQLite. Локальный LLM по умолчанию выключен — мобильные устройства проксируют запросы к серверу или к собственному desktop-узлу пользователя через VPN/WireGuard. |

## 1.1. Ключевое архитектурное предупреждение по Tauri mobile

Tauri 2.x формально стабилен, но команда Tauri явно зафиксировала: «mobile is not first-class citizen yet». Не все desktop-плагины портированы на iOS/Android, и WKWebView-обёртка может попасть под App Store Review 4.2, если приложение не предоставит существенный нативный слой. Из этого следуют три практических решения:

1. Мобильная версия проектируется как «augmented WebView» — поверх Next.js-фронта обязательно идут нативные слои: безопасное хранилище ключей, push-уведомления, share-extension, биометрия, speech-to-text. Это закрывает требование «не просто веб-обёртка».
1. Локальный LLM на мобильных вне scope первых релизов. Вместо этого мобильное приложение проксирует «private»-категории либо к серверу пользователя, либо к собственному десктоп-узлу через защищённый канал.
1. Веб- и десктоп-версии — основные. Мобильная — для quick capture, чтения, дневников, голосовых заметок, базовой работы с задачами и квестами. Тяжёлый исследовательский режим и графовая визуализация — на десктопе.

# 2. Local-first гибридная модель

Принцип: пользователь всегда может работать с собственными записями локально и оффлайн; сервер — координатор, а не единственный держатель истины. При этом тяжёлые серверные компоненты (мультиагентный пайплайн, графовая БД, эмбеддинги для всего корпуса) остаются на сервере.

## 2.1. Что хранится локально

- Полная зеркальная копия Markdown-корпуса пользователя на десктопе. Каталог наблюдаем file-watcher'ом.
- Подмножество корпуса на мобильном — последние N дней + закреплённые заметки + активные проекты. Полный корпус доступен по запросу.
- Локальный SQLite-кэш с метаданными, индексами и недавними эмбеддингами для оффлайн-поиска.
- Очередь исходящих изменений (outbox) — все локальные правки, ожидающие синхронизации.
- Конфигурация: ключи API, настройки приватности, ключ vault-шифрования (только локально, никогда не уходит на сервер).

## 2.2. Что остаётся на сервере

- Postgres (Supabase) — пользователи, права, индексы, метрики, привычки, квесты, RPG-данные, история версий, embeddings для всего корпуса (pgvector).
- TypeDB — онтология и граф знаний. Не реплицируется на клиент — слишком тяжёлый и связный.
- S3 (MinIO) — каноническая копия .md и медиа. Object Versioning включён.
- NestJS + Mastra.ai — мультиагентный конвейер, тяжёлый синтез, фоновые джобы.
- Очереди (Supabase Queues / pgmq) — асинхронные обработки, ретраи, расписание.

## 2.3. Стратегия синхронизации

Базовая модель — eventual consistency с серверным авторитетом и поэтапным повышением сложности конфликтного разрешения.

1. Уровень 1 (MVP): last-write-wins на уровне файла. Каждый .md имеет updated_at и hash. При расхождении побеждает более поздняя версия; «проигравшая» версия не теряется — сохраняется в истории с пометкой conflict-suspended.
1. Уровень 2: текстовый 3-way merge для незаблокированных файлов, основанный на base-версии (последняя синхронизированная).
1. Уровень 3 (опционально, для активной коллаборативной работы): Yjs/Automerge CRDT поверх отдельных файлов с присутствием через Supabase Realtime. Только для файлов, помеченных как collaborative.
Realtime-канал на Supabase используется для: (а) живых уведомлений о завершении тяжёлых агентских задач, (б) presence — кто из устройств пользователя онлайн, (в) push новых производных артефактов (insights, quests, digests).

## 2.4. Категории видимости и маршрутизация по моделям

| **Категория** | **Что это** | **Маршрутизация LLM** |
| --- | --- | --- |
| default | Большинство заметок: исследования, проекты, общие идеи | Server-side через Anthropic API (Claude Sonnet/Haiku/Opus) |
| private | Психотерапевтические дневники, личные финансы, отношения | Только локальный Ollama на десктопе (Qwen/Llama). Мобильный проксирует к desktop-узлу пользователя или вообще отказывает в обработке |
| vault | Самое чувствительное (медицинские, юридические, политически рискованные) | Дополнительное E2E-шифрование пользовательским ключом. LLM — только локальный Ollama; сервер хранит лишь шифротекст и не имеет ключа |

Категория назначается пользователем явно или агентом Domain Classifier автоматически (с подтверждением для чувствительных). Изменение категории — это аудируемое событие.

# 3. Серверный стек

## 3.1. NestJS + @mastra/nestjs

Серверная часть — NestJS-приложение, в которое Mastra.ai встраивается через официальный адаптер @mastra/nestjs. Адаптер обеспечивает:

- Регистрацию Mastra-агентов и инструментов внутри NestJS-модулей с поддержкой DI.
- Доступ к Mastra-instance, request context, abort signal через middleware на каждом запросе.
- Прокидывание auth-хука: единый guard NestJS пускает запрос дальше или режет.
- Stream-ответы (SSE / Vercel AI SDK) из контроллеров NestJS.
- MCP transport для общения с MCP-серверами (внешние коннекторы пользователя).
- Rate limiting и graceful shutdown.
Структура NestJS-модулей повторяет уровни агентов из v1 концепции: ingestion.module, understanding.module, linking.module, memory.module, synthesis.module, therapy.module, habits.module, projects.module, metrics.module, research.module, gamification.module, orchestration.module. Каждый модуль содержит NestJS-сервисы для бизнес-логики и Mastra-агенты, которые внедряются как зависимости.

## 3.2. Supabase: что используется и почему

Supabase выбирается как единый backend-слой с обязательным требованием — все используемые компоненты должны работать в self-hosted сборке через официальный docker-compose. Это политика «no vendor lock».

| **Сервис Supabase** | **Self-host** | **Назначение в Nabu** |
| --- | --- | --- |
| Postgres (с pgvector) | Да | Основная БД. Все метаданные заметок, пользователи, права, временные ряды метрик, история. Эмбеддинги — в pgvector |
| GoTrue (Auth) | Да | Аутентификация: email/password, OAuth (Google, GitHub, Apple), magic link. RLS-политики на основе auth.uid() |
| Realtime | Да | WebSocket-канал для уведомлений об агентских задачах, presence устройств, опционально CRDT-синхронизация коллаборативных файлов |
| Storage (S3-compat) | Да | Каноническая копия .md и медиа. Object versioning. S3 API для внешних бэкапов и для прямого доступа из агентов |
| Edge Functions (Deno) | Да | Лёгкие веб-хуки и эндпоинты, не требующие NestJS-инфраструктуры: оплата (если когда-нибудь будет), приём входящих писем, telegram-bot endpoints |
| Queues (pgmq) | Да | Все асинхронные задачи: транскрипция, эмбеддинг, тяжёлые синтезы, импорт корпусов, ретраи |
| pg_cron | Да | Расписание: ежедневные дайджесты, обработка стриков, ночная переиндексация, агрегация метрик |
| Logflare/Analytics | Опционально | Логи. Можно отключить и заменить на собственный стек (Loki/Grafana) |

Все клиентские взаимодействия с Postgres проходят через Row Level Security. Для Queues пользователю напрямую обращаться к pgmq_public не нужно — все постановки задач инкапсулированы в NestJS-контроллерах с явной авторизацией.

## 3.3. TypeDB

TypeDB 3.0 как полиморфный гиперграф онтологии остаётся ключевым серверным компонентом — без замены на Neo4j или RDF. Причины: native-поддержка многоарных отношений, иерархия типов и атрибутов, правила вывода, которые читаются как декларативные утверждения и хорошо ложатся на промпты агентов.

TypeDB живёт в отдельном Docker-контейнере, доступна через TypeDB Driver Node.js. Backup TypeDB — это экспорт TQL-схемы и data dump'а, его делает отдельный pg_cron-task раз в сутки.

## 3.4. LLM-маршрутизатор

Mastra.ai поддерживает многопровайдерный model router (3000+ моделей через единый интерфейс). Nabu использует это для маршрутизации на разные модели по характеру задачи:

- Claude Haiku 4.5 — Triage, Entity Extractor, Domain Classifier, Intent Detector (лёгкие классификаторы, низкая стоимость).
- Claude Sonnet 4.6 — основная рабочая лошадка: Linker, Insight Agent, CBT/Gestalt-агенты, Coach, Project Manager, Quest Master.
- Claude Opus 4.7 — Document Synthesizer, Critic, Hypothesis Agent, годовые/квартальные дайджесты — задачи с высокой ставкой на качество синтеза.
- Локальный Ollama (Qwen 2.5 / Llama 3.3) — private/vault категории. Mastra обращается к Ollama через стандартный OpenAI-совместимый интерфейс.
Выбор провайдера — конфигурация на уровне агента, не хардкод. Пользователь может в настройках перевести любой агент на локальную модель, приняв компромисс в качестве.

# 4. Клиентский стек

## 4.1. Frontend

- Next.js (App Router) + TypeScript + bun + Tailwind.
- CopilotKit — для генеративного UI, side-bar чата с агентами, slash-команд, динамической перестройки интерфейса.
- Один и тот же фронтенд-код собирается тремя способами: (а) обычный Next.js-сервер для веб-версии; (б) статическая сборка SPA, упакованная в Tauri-desktop; (в) та же SPA-сборка с адаптивной разметкой и mobile-плагинами Tauri для iOS/Android.

## 4.2. Локальное хранилище на десктопе

- Tauri Filesystem plugin — для работы с .md-файлами в локальном корне vault'а пользователя. Структура зеркалит структуру в S3.
- Tauri SQL plugin (sqlx) — локальный SQLite с теми же метаданными, что в серверном Postgres (подмножество). Используется для оффлайн-поиска, ускорения навигации, очереди исходящих изменений.
- Tauri Store plugin — настройки приложения и user preferences.
- Tauri Stronghold plugin — зашифрованное хранилище ключей: токены API, vault-encryption key.
- Tauri Updater — самообновление приложения.

## 4.3. Локальный LLM (Ollama)

На десктопе пользователь может (опционально) запустить Ollama локально. Nabu обнаруживает его автоматически через стандартный порт и предлагает использовать для категорий private/vault.

- Рекомендуемые модели: Qwen 2.5 Coder 7B / 14B для технических задач, Qwen 2.5 14B-Instruct для общих задач, Llama 3.3 70B при достаточных ресурсах GPU.
- Локальный Whisper для транскрипции голосовых заметок (через Tauri shell-команду или встроенный whisper.cpp).
- Локальные эмбеддинги через nomic-embed-text или bge-m3 — для приватных данных, чтобы они не уходили на сервер даже в виде вектора.

## 4.4. Локальное хранилище на мобильном

На мобильном — урезанная версия desktop-схемы:

- Локально хранятся только закреплённые и недавние заметки (rolling window ~30 дней + ручные pin).
- Локальный LLM отключён по умолчанию (см. раздел 1.1). Для private категории мобильное приложение запрашивает обработку у пользовательского desktop-узла через защищённый канал или показывает уведомление «обработка отложена до доступа к desktop».
- Quick capture: голос → транскрипция через нативные iOS Speech / Android SpeechRecognizer (без отправки на сервер до явного подтверждения).
- Share extension: возможность отправить в Nabu любой текст/ссылку/изображение из других приложений.

# 5. Сквозной поток данных при пользовательском вводе

Чтобы убедиться, что архитектура работает целиком, рассмотрим путь типичной заметки от ввода до индексации и обогащения.

## 5.1. Сценарий: пользователь записал заметку на мобильном

1. Пользователь делает quick capture в Nabu mobile (текст или голос). Если голос — выполняется локальная транскрипция.
1. Заметка немедленно сохраняется локально как .md с YAML frontmatter, попадает в outbox-очередь.
1. При наличии связи outbox синхронизируется: PUT в Supabase Storage (.md-файл) и INSERT в Postgres (метаданные). Транзакция координируется через NestJS API.
1. NestJS отправляет сообщение в Supabase Queue inbox.created.
1. Воркер (фоновый процесс или Edge Function) забирает сообщение и инициирует мультиагентный конвейер через Mastra.ai.
1. Conductor решает маршрут. Для обычной заметки: Scribe → Triage → Entity Extractor → Domain Classifier → Affect Analyzer → Intent Detector (часть из них может выполняться параллельно).
1. Linker через pgvector ищет близкие заметки; через TypeDB ищет связи в графе.
1. Ontology Maintainer при необходимости расширяет TypeDB-схему; Taxonomy Curator обновляет адаптивную таксономию.
1. Librarian определяет статус заметки (fleeting/literature/evergreen).
1. Insight Agent (если ввод задевает паттерн) формирует наблюдение. RPG Game Master начисляет XP.
1. Critic просматривает все производные артефакты.
1. Результаты записываются в Postgres + S3; через Supabase Realtime отправляется push на все устройства пользователя — обновлённые backlinks, новые insights, обновлённый стат, обновлённый граф.
1. Mobile получает push, обновляет локальную SQLite-кэш и UI.

## 5.2. Сценарий: запись помечена как private

1. Заметка ввоодится на desktop. Пользователь явно ставит visibility: private или Domain Classifier помечает её таковой (например, КПТ-дневник).
1. Заметка сохраняется локально в .md.
1. На сервер уходит только зашифрованный blob (для vault) или только метаданные без содержимого (для private — содержимое не уходит вообще).
1. Мультиагентный конвейер запускается локально на desktop через Ollama. Mastra.ai в этом сценарии работает в режиме «локальный сервер» внутри Tauri-процесса.
1. Результаты обработки (онтологические связи, эмбеддинги, инсайты) тоже хранятся локально и не реплицируются на сервер.
1. Опция: пользователь может вручную «продвинуть» производный артефакт (например, обезличенный инсайт) на сервер.

## 5.3. Сценарий: пользователь оффлайн

1. Все правки идут в локальный outbox.
1. Доступны все локальные функции: чтение, редактирование, поиск (по локальному SQLite + локальным эмбеддингам).
1. Тяжёлая ИИ-обработка ставится в pending: если есть Ollama — выполняется локально с пометкой preliminary; иначе ждёт сервера.
1. При восстановлении связи outbox дренируется, серверный конвейер дообрабатывает то, что было preliminary.

# 6. Аутентификация, права, безопасность

- Auth — Supabase GoTrue: email/password, OAuth (Google, GitHub, Apple), magic-link, опционально TOTP MFA.
- Авторизация — RLS в Postgres + декораторы guard'ов в NestJS. Каждый запрос к серверным агентам проходит через NestJS AuthGuard, который валидирует JWT и устанавливает userId в Mastra runtime context.
- End-to-end шифрование vault — ключ деривуется из пароля пользователя на клиенте (Argon2id), ключ никогда не уходит на сервер. Сервер хранит только шифротекст.
- Audit log — отдельная таблица в Postgres с журналом всех агентских вызовов и каких данных в каком объёме уходило на внешние API. Доступна пользователю в любой момент.
- Бэкап и экспорт — одна команда выгружает всю информацию пользователя как директорию .md + JSON-дампы метаданных и онтологии.
- Удаление — настоящее: purge из Postgres, S3, pgvector, TypeDB и принудительное истечение всех S3-версий. RTBF-готово.

# 7. Адаптация под AI-команду разработки (Aperant и аналоги)

Документация Nabu должна быть пригодна для машинного чтения и автономного исполнения AI-coding-агентами через инструменты типа Aperant (autonomous multi-session AI coding). Это диктует особый стиль всех последующих документов.

## 7.1. Принципы документации, ориентированной на AI-исполнителей

1. Атомарность требований. Каждое функциональное и нефункциональное требование — отдельное утверждение с уникальным ID (см. SRS). Никаких параграфов, в которых перемешано несколько требований.
1. Уникальные идентификаторы. Сквозная нумерация: эпики E-NN, фичи F-NN.NN, истории US-NN.NN.NN, требования FR-NNNN / NFR-NNNN. Ссылки между документами — только по ID, не «как сказано выше».
1. Acceptance criteria в формате Given-When-Then на каждую user story. AI-агент должен понимать критерий приёмки без интерпретации.
1. Definition of Done на каждый эпик — что считается завершённым. Включает тесты, документацию, наблюдаемость.
1. Технические допущения и ограничения — отдельным списком. AI-агент не должен «угадывать», какие версии библиотек и какие архитектурные паттерны разрешены.
1. Traceability matrix — каждая user story ссылается на функциональные требования из SRS; каждая фича — на эпик; каждое требование — на как минимум одну user story.
1. Тестируемость. Каждое требование должно быть верифицируемо автоматически: unit-тесты, интеграционные тесты, e2e-тесты, evals (для агентов).
1. Семантические шаблоны. Все user stories строятся по единому шаблону «As a [роль], I want [цель], so that [польза]» с явными констрейнтами по полям.

## 7.2. Discipline на этапах разработки

- Каждая user story исполняется в изолированной git worktree (Aperant поддерживает это нативно).
- До начала исполнения AI-агент обязан зачитать связанные FR из SRS и проверить, что задача укладывается в Definition of Done эпика.
- Каждый PR проходит автоматическую проверку: lint, type-check, unit-test, интеграционные тесты, eval-тесты для агентов.
- Human review — обязателен для всего, что касается psychotherapy-агентов, безопасности, биллинга, удаления данных.

## 7.3. Состав комплекта документации

Этот концептуальный документ — точка входа. Остальные документы в комплекте:

1. «Функциональная матрица» — бенчмарк существующих инструментов и целевая матрица Nabu с разбиением на этапы.
1. «SRS по ISO/IEC/IEEE 29148:2018» — функциональные и нефункциональные требования с ID, описанием, источником, критериями верификации.
1. «Agile backlog» — эпики, фичи, user stories с acceptance criteria, готовые к подаче в Aperant.
Последующие документы (могут быть добавлены позже): SAD (Software Architecture Description) по ISO/IEC/IEEE 42010, ADR (Architecture Decision Records), Test Strategy, AI Agent Evaluation Suite, Data Privacy Impact Assessment, Operational Runbook.
