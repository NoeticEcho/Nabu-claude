# Zero-config запуск Nabu

Nabu ставится и запускается **одной командой** — без ручной настройки БД, ключей и моделей.

```bash
curl -fsSL https://raw.githubusercontent.com/noeticecho/nabu-claude/master/scripts/install.sh | bash
```

Или из checkout: `bash scripts/install.sh`. После установки доступна команда **`nabu`**.

## Что делает установка

1. **Проверяет** git, Node ≥22, npm, Docker (+compose v2), claude CLI. Без Docker продолжит
   с предупреждением (без Docker Nabu не работает — стек обязателен).
2. **Клонирует/обновляет** репозиторий (`~/.nabu-claude` по умолчанию), `npm install && build`.
3. **Линкует** `nabu` в `~/.local/bin`.
4. **`nabu init`** — вся инфраструктура сама:
   - `.env` генерируется (пароли — случайные; существующие ключи никогда не перезаписываются);
   - **порты выбираются свободные** (стек не столкнётся с вашими postgres/typedb/ollama);
   - Docker-стек: **pgvector (pg17) + TypeDB 3.4 + Ollama** (последний — только если нет
     локального бинарника: при наличии host-ollama init использует ЕГО и качает модель в его
     `~/.ollama` — это осознанное переиспользование; всё слушает только `127.0.0.1`);
   - схемы Postgres `000–008` применяются идемпотентно; `000_standalone_bootstrap.sql`
     создаёт совместимый поднабор доменных таблиц основного Nabu + дефолтного пользователя
     (`NABU_USER_ID`); `008_standalone_research_notes.sql` — заметки (`notes`/`note_links`,
     schema/typedb–2.3), research-слой (`sources`/`claims`/`claim_relations`) и колонки
     `level`/`tuppi`/`parent_goal_id` — так `nabu-domain`/`nabu-analytics` и research-агенты
     работают полностью без облака;
   - TypeDB: база + **полная онтология** через HTTP API в явном порядке:
     `000_standalone_base.tql` (concept) → `memory.tql` (память/ассоциации) →
     `002_standalone_domain.tql` (вся доменная онтология schema/typedb: person/place/project/
     goal/habit/idea/note/decision/emotion/metric/quest/source-entity + связи related/
     depends-on/supports/contradicts/achieves/blocks/derives-from). При неудаче —
     Postgres-fallback, не блокирует;
   - модель эмбеддингов `nomic-embed-text-v2-moe` докачивается через Ollama API;
   - финальный smoke-тест (Postgres + Ollama + память + TypeDB).

| Режим | Когда | Что с Docker |

## Демон

```bash
nabu start      # фон: планировщик + TTL-purge + проверка обновлений + веб-чат
nabu status     # демон, docker-сервисы, ollama, обновления, расписание
nabu logs       # хвост лога демона
nabu stop       # остановить демон (--infra — и docker-стек)
nabu install-service  # автозапуск через systemd user unit (Linux)
```

Демон решает проблему **запуска агентов по расписанию**: задачи из `config/schedule.json`
исполняются как `claude -p "<команда>"` headless в workspace `~/nabu` с узким allowlist
инструментов (только MCP Nabu + чтение + Task). По умолчанию все agent-задачи **выключены**
(они тратят квоту Claude) — включаются осознанно:

```bash
nabu schedule                # список
nabu schedule enable digest  # еженедельный дайджест в 09:00
```

Также демон каждый час чистит истёкшую рабочую память (TTL) и раз в сутки проверяет
обновления GitHub.

**Семантика задач (честно):** запуск фиксируется ДО выполнения (anti-poison), поэтому задачи —
*at-most-once*: если демон остановлен/перезапущен во время выполнения, результат этого запуска
теряется, повтор — в следующее окно `everyDays`. Итог задачи ограничен ~2МБ (больше — без push,
только лог). При остановке демона запущенные claude-процессы задач завершаются самостоятельно
(их результат не записывается). **Итоги задач проактивно отправляются в Telegram** (если бот настроен;
`push:false` у задачи отключает) и сохраняются в `.nabu/job-results.json`.
Задача `triage` (`/nabu-triage`) ежедневно разбирает «Входящие»: заметки из Telegram +
файлы `00-inbox` → классификация → сущности/факты/граф → статусы. `backup` — internal-задача
(локальная, без Claude).

## Веб-чат

```bash
nabu chat   # http://127.0.0.1:4517 (поднимет демон при необходимости)
```

Лёгкий веб-интерфейс (без зависимостей и сборки), который демон отдаёт сам:
- треды с историей (история — в браузере, на сервере только метаданные);
- потоковый ответ (SSE) + индикация вызова инструментов;
- **агенты исполняются в Claude Code**: каждый запрос — headless-сессия
  (`claude -p --output-format stream-json --resume <session>`), т.е. полный
  адъютант/Совет/память, та же кодовая база, продолжение контекста между сообщениями;
- слушает только `127.0.0.1`; allowlist инструментов узкий (write-операции файлов недоступны).

Почему не Next.js/CopilotKit: отдельный build-pipeline и сотни МБ зависимостей против
«запустил и работает». Поверх того же API (`/api/chat`, `/api/threads`) богатый фронт
можно добавить позже, не меняя демона.

## Статистика

- **В чате**: кнопка «📊 Статистика» — дашборд: память (эпизоды/факты/нарративы/рабочая/намерения),
  знания (документы/чанки), граф (концепты/связи), Совет (совещания/позиции/советы/follow-up),
  сферы жизни (проекты/задачи/цели/привычки/квесты), RPG (XP по 8 атрибутам), система,
  динамика за 14 дней. Автообновление раз в 60с; при недоступности БД — «н/д», чат не ломается.
- **В терминале**: `nabu stats` — тот же обзор + спарклайн за 14 дней.
- API: `GET /api/stats` (кэш 5с; `?fresh` — принудительно).

## Логи (для диагностики и самопочинки)

| Что | Где | Формат |
|---|---|---|
| Демон (планировщик, purge, обновления, чат-сервер) | `~/nabu/.nabu/daemon.log` | строки с ISO-временем |
| Обмены веб-чата | `~/nabu/.nabu/logs/chat.jsonl` | JSONL: `{ts, evt, threadId, ms, costUsd, ok}` — **текст сообщений не пишется** (приватность); ротация 5МБ |
| Scheduled-задачи | `~/nabu/.nabu/job-<name>.log` | вывод headless-запуска |

Чтение: `nabu logs` (демон) · `nabu logs --chat` · `nabu logs --job digest` · `--n=200`.
Эти логи структурированы так, чтобы Claude Code мог их прочитать и починить проблему в коде.

## Telegram-клиент (опционально)

Второй клиент для ежедневного использования с телефона. Включение:
1. Создайте бота у @BotFather → токен в `.env`: `TELEGRAM_BOT_TOKEN=...`
   (опц. `TELEGRAM_CHAT_ID=` — жёсткая привязка к чату). Перезапустите демон (`nabu stop && nabu start`).
2. Создайте **приватную форум-группу** (супергруппа с темами), добавьте бота **администратором**
   (право управления темами), отправьте `/start`, затем `/setup` — бот сам создаст темы:
   - **📥 Входящие** — скидывайте заметки/мысли: сохраняются в таблицу `notes` (private);
   - **🎖 Адъютант** — основной диалог (тот же пайплайн, что веб-чат: память, Совет, resume);
   - **темы 9 министров** (🏥 Здоровье … 📋 Дела) — вопрос напрямую конкретному министру.
   **Голосовые сообщения** (voice/audio/video note) расшифровываются ЛОКАЛЬНО (whisper,
   `scripts/transcribe.py`) и маршрутизируются как текст той же темы.
3. Однопользовательская привязка: бот отвечает только в привязанном чате; чужие — игнор.

⚠️ **Приватность**: содержимое сообщений транзитом проходит через серверы Telegram — это ваш
осознанный выбор канала. `vault`-содержимое туда не отправляйте. Логи бота — без текста сообщений.

## Приватность уровня vault (E2E)

`visibility: vault` — максимальный уровень: содержимое шифруется **до записи в БД**
(AES-256-GCM, ключ `NABU_VAULT_KEY` генерируется `nabu init` и живёт только в вашем `.env`).
Эмбеддинги для vault НЕ считаются (даже локально) — векторный поиск по vault невозможен
намеренно. Чтение — только явное (`nabu-memory.list_vault`). Разбор vault-заметок — локальной
LLM (`nabu-pipeline.extract_entities_local({noteId})`): текст не попадает в контекст Claude.
⚠️ Потеря `NABU_VAULT_KEY` = безвозвратная потеря vault-записей — храните копию .env.

## Подтверждения высокорисковых действий (вне модели)

Запросы `request_approval` теперь подтверждает ЧЕЛОВЕК кнопками — модель не может одобрить
сама себя: в веб-чате карточка «🔐 Подтверждения» (✅/❌), в Telegram — /approvals с
inline-кнопками. Решение пишется как `user:web` / `user:telegram` в аудит.

## Локальная LLM для приватного (опционально)

`env NABU_LOCAL_LLM` (по умолчанию `qwen3:4b`; `ollama pull qwen3:4b`) — маленькая локальная
модель для экстракции сущностей/фактов из private/vault-контента
(`nabu-pipeline.extract_entities_local`). Текст обрабатывается целиком на машине.

## Бэкап

`nabu backup [--out=dir]` — три архива в `~/nabu/.backups` (ретенция 7):
Postgres (`pg_dump`→gzip), TypeDB (кратк. остановка → tar тома), workspace (tar).
Ежедневно по расписанию: `nabu schedule enable backup`. Восстановление: pg — `gunzip | psql`;
typedb — tar в том; workspace — `tar -xzf`.

## Обновления

```bash
nabu update    # git pull --ff-only → npm install → build → рестарт демона
```

Демон ежедневно проверяет обновления и показывает их в `nabu status`. Полностью
автоматическое применение — opt-in: `"auto_update": true` в `config/schedule.json`.

## Диагностика

```bash
nabu doctor    # node/docker/claude/.env/сборка/ollama/модель/smoke — с кодом выхода
```

## Локальный мозг (экспериментально, T1)

При недоступном Claude (`NABU_OFFLINE_FALLBACK=1`) чат теперь отвечает **агентным циклом локальной
модели с реальными инструментами** (cli/local-brain.mjs): память (recall/remember), задачи
(list/add/update), календарь, знания — через те же MCP-серверы. Высокорисковое (вебхуки, vault,
approvals) локальному мозгу не выдаётся осознанно. Модель — `NABU_LOCAL_LLM` (ollama);
инструменты — ядро из 8 (расширение `NABU_BRAIN_TOOLS=csv`), таймаут шага —
`NABU_BRAIN_STEP_TIMEOUT_MS`. На CPU ответ занимает минуты — это резерв, не замена; полноценный
«локальный мозг» гейтируется прогоном evals Совета: `node evals/runner.mjs --mode live --brain
local` (см. ROADMAP). При сбое цикла — тихий откат к простому recall-ответу.

## Где живут ваши настройки

Рантайм-конфиги — в **`~/nabu/.nabu/config/`** (`schedule.json`, `profiles.json`,
`integrations.json`, `nabu.config.json`): `nabu init` сеет их из шаблонов репозитория, дальше
правьте живые копии — обновления (`nabu update`) их не трогают и git-конфликтов не создают.
`NABU_CONFIG_DIR` переопределяет путь. Файлы в `config/` репозитория — только шаблоны.

## Обучаемая проактивность

Под каждым проактивным push'ем (брифинг, напоминания, обзоры) — кнопки **👍 Полезно / 🔇 Реже**:
🔇 удваивает интервал джобы (до ×4), 👍 возвращает исходный ритм. Без реакции 3 push'а подряд —
интервал удваивается сам (Nabu затихает, а не спамит). Состояние — `~/nabu/.nabu/proactivity.json`;
выключить джобу совсем — `nabu schedule disable <имя>`. Ритуалы масштаба: джобы `quarterly-review`
и `yearly-review` (Claude-квота, выключены по умолчанию) пишут главы автобиографии.

## Профили (мульти-пользователь, v1)

Создание: **`nabu profiles add anna`** — команда заводит пользователя в БД и пишет валидный
профиль (namespace **и** user_id — обязательны оба; неполный профиль отклоняется fail-closed,
иначе часть данных — задачи/финансы/здоровье — утекала бы в основное пространство).
Любая команда с `--profile=anna` (или `NABU_PROFILE=anna`) работает в её пространстве — память,
задачи, заметки, финансы, approvals полностью раздельны. Один демон = один профиль; второй
одновременный профиль = второй `NABU_HOME` + порт. Список — `nabu profiles`. **Граф знаний (TypeDB) общий между профилями** — онтология не
профилируется (осознанно: понятия мира общие; личные данные живут в памяти/домене, которые
изолированы).

## Шифрованный бэкап

`nabu backup --encrypt` — каждый архив шифруется AES-256-GCM ключом `NABU_VAULT_KEY`
(формат `.enc`: NBK1+IV+шифртекст+tag), plaintext удаляется. Такой архив можно класть на любые
узлы (Syncthing/rsync/облако) — без ключа он нечитаем. Восстановление:
`nabu backup-decrypt <файл.enc> [куда]`; полный цикл — **`nabu restore <каталог>`** (сам
расшифрует .enc, корректно заменит БД/том/workspace с подтверждением; заливка дампа в живую
БД руками даёт конфликты — не делайте так). ⚠ Потеря `NABU_VAULT_KEY` = потеря зашифрованных бэкапов.

## Несколько устройств

Nabu — standalone: память живёт в docker-томах одной машины. Телефон подключается к домашнему
демону: Telegram-бот или PWA веб-чата (Tailscale/WireGuard до `127.0.0.1:4517` — наружу чат не
биндится). Перенос на новый компьютер: `nabu backup [--encrypt]` → перенести архивы →
`nabu init` → восстановить одной командой: `nabu restore <каталог-бэкапа>`;
`NABU_VAULT_KEY` из старого `.env` перенести обязательно. Одновременная работа двух машин с
одной памятью не поддерживается — сознательное упрощение v1.0.0.

## Сброс и удаление

Данные живут в named volumes `nabu-pgdata`, `nabu-typedb`, `nabu-ollama`.

```bash
nabu reset [--hard] [--dry-run] [--yes]
```
Стирает **данные и state** (docker-тома → вся память/vault/история; `~/nabu/.nabu`), но
сохраняет установку, **контент workspace**, `.backups` и `.env` (ключи!). `--hard` дополнительно
удаляет `.env` — NABU_VAULT_KEY будет потерян и vault в бэкапах станет нерасшифруемым. После —
`nabu init && nabu start` для чистого старта.

```bash
nabu uninstall [--purge-workspace] [--images] [--dry-run] [--yes]
```
Полное удаление: демон, автозапуск (systemd/launchd/schtasks), docker-стек с томами,
CLI-обёртка, state. По умолчанию **сохраняет** workspace `~/nabu` (ваши файлы + бэкапы) и
`.env`; `--purge-workspace` удаляет и их (двойное подтверждение). `--images` — удалить и
docker-образы (осторожно: могут использоваться другими проектами). Репозиторий не удаляет
сам себя — команда печатается в конце. Обе команды показывают план и требуют подтверждения
(`--yes` для скриптов, `--dry-run` — только план).

## Утренний брифинг и голосовые ответы

**Брифинг** — `internal`-джоба (детерминированная, **без Claude** и без расхода квоты): собирает
погоду (коннектор `weather`), задачи и привычки на день, ожидающие намерения и события из
календаря (ICS) и пушит одним сообщением в Telegram. Включается в `config/schedule.json`:

```json
{ "name": "briefing", "internal": "briefing", "at": "08:00", "everyDays": 1, "enabled": false, "push": true }
```

По умолчанию `enabled: false` — включить: `nabu schedule enable briefing`. Ничего не отправляется
в облако; всё считается локально из вашей БД и конфигов.

**Голосовые ответы** (TTS): `NABU_TTS=1` + `pip install piper-tts` (для формата голосовых заметок
Telegram — ещё `ffmpeg` в PATH, тогда отдаётся `.ogg/opus`, иначе обычный `.wav`). После этого
ответы адъютанта в Telegram приходят **голосом** (`scripts/tts.py`, синтез локальный — текст не
уходит в облако). `NABU_TTS_VOICE` выбирает голос (по умолчанию `ru_RU-irina-medium`; скачивается
один раз в `~/.nabu/tts-voices`). `NABU_TTS_MAX_CHARS` (по умолчанию 900) — длиннее озвучивается
только вступление, полный текст остаётся в сообщении.

## Индексация PDF и сканов

`/nabu-index` (инструмент `index_folder`) кроме `.md/.txt` умеет **PDF** и **изображения**
(`.png/.jpg/.jpeg/.webp/.tiff`) — договоры, чеки, сканы. Извлечение текста **строго локальное**,
в облако ничего не уходит:

- **PDF** → `pdftotext -layout` (пакет **poppler**);
- **изображения / сканы** → OCR через `tesseract`.

Установка бинарей:

```bash
# Debian/Ubuntu
sudo apt install poppler-utils tesseract-ocr tesseract-ocr-rus
# Arch
sudo pacman -S poppler tesseract tesseract-data-rus tesseract-data-eng
# macOS
brew install poppler tesseract tesseract-lang
```

Языки OCR задаются `NABU_OCR_LANGS` (по умолчанию `rus+eng`; формат tesseract — коды через `+`).

**Честная деградация:** если бинаря нет (`pdftotext`/`tesseract` не установлен) или извлечение дало
пустой результат (например, отсканированный PDF без текстового слоя, а tesseract не установлен) —
такой файл **пропускается** с указанием причины, а индексация остальных файлов **продолжается**.
Причины пропусков возвращаются в результате инструмента (`skipped: [{file, reason}]`, счётчик +
первые 5) и печатаются в лог.
