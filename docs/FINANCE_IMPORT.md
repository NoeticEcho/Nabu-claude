# Импорт финансов

Модуль импорта финансов в Nabu построен вокруг одного принципа: **файловые выписки банков,
разбираемые локально**. Никаких банковских API, никакого OAuth, никаких токенов доступа к счёту.

## Философия: файлы-выписки, а не API банков

Банковские API либо закрыты для частных лиц (доступ только партнёрам по договору), либо требуют
OAuth-потока, при котором данные о ваших счетах проходят через сторонний сервер. Это ровно то,
чего инвариант приватности Nabu не допускает: **финансы по умолчанию `private`**.

Зато **экспорт выписки в CSV есть в личном кабинете почти любого банка**. Вы выгружаете файл, а
Nabu разбирает его **на 100% локально** и складывает транзакции в общую таблицу
`finance_transaction`. Данные никуда не уходят — ни в облако, ни в Claude, ни в логи.

## Как выгрузить CSV

- **Тинькофф**: личный кабинет / приложение → счёт → «Выписка» / «Операции» → период → экспорт в
  **CSV** (разделитель `;`, десятичная запятая, дата `ДД.ММ.ГГГГ`). Формат распознаётся автоматически.
- **Сбербанк (СберБанк Онлайн)**: История операций → выписка за период → выгрузка в CSV/таблицу.
  Если банк отдаёт только PDF/XLS — пересохраните как CSV (Excel/Numbers → «Сохранить как CSV»).
- **Любой другой банк / generic**: подойдёт любой CSV, где в заголовке есть колонки даты и суммы.
  Разделитель (`,` или `;`) и десятичная запятая определяются автоматически.

Минимально нужны две колонки: **дата** и **сумма**. Описание, валюта и категория — опциональны.

## Команда

```
nabu import-finance <файл.csv> [--source bank]
```

(CLI-обвязка подключается в основной сессии.) Что делает импорт:

- **Автоопределение**: разделитель `,`/`;`; десятичная запятая («1 234,56» → 1234.56, пробелы и
  неразрывные пробелы как разделители тысяч отбрасываются); формат даты `ДД.ММ.ГГГГ` и `ГГГГ-ММ-ДД`.
- **Сопоставление колонок** по подстроке заголовка, без учёта регистра, RU и EN:
  - дата: `дата операции` · `дата` · `operation date` · `date`
  - сумма: `сумма операции` · `сумма` · `amount` (**расход отрицательный, приход положительный**)
  - описание: `описание` · `назначение` · `description` · `merchant` · `категория банка`
  - валюта: `валюта` · `currency` (по умолчанию `RUB`)
  - категория банка: `категория` · `category` (если банк её дал — она в приоритете)
- **Вывод**: `inserted` (новых), `deduped` (уже были), разбивка `byCategory` (суммы по модулю) и
  **сводка за 30 дней** (расход / доход / топ-5 категорий трат / число операций).

Повторный импорт того же файла ничего не дублирует: дедуп по `sha256(дата|сумма|описание)`
(уникальный индекс `user_id + tx_hash`). Можно спокойно реимпортировать перекрывающиеся периоды.

## Категоризация

Если банк не дал категорию, Nabu проставляет её по **встроенным правилам** (~15 категорий,
совпадение по ключевым словам в описании, RU/EN, первое совпадение выигрывает):

`продукты` · `кафе` · `транспорт` · `жильё` · `связь` · `здоровье` · `подписки` · `одежда` ·
`техника` · `развлечения` · `путешествия` · `образование` · `переводы` · `зарплата` · `другое`.

Категория из банка (если есть в выписке) **всегда приоритетнее** встроенных правил.

## Что дальше

- Данными пользуется **министр finance**: реальные траты вместо догадок.
- Вопросы в свободной форме: `/nabu-ask «сколько я трачу на кафе»`, «какой был доход в июне».

## Честные ограничения

- Правила категоризации покрывают **~15 частых категорий** и типовые РФ/EN-мерчанты — редкие или
  локальные названия попадут в `другое`. Категорию всегда можно поправить вручную в БД.
- Банковские CSV-форматы **не стандартизированы**: экзотическая выписка может не распознаться.
  Импорт не падает — непонятные строки идут в **warnings** счётчиком. **Проверяйте warnings** после
  импорта: если пропущено много строк, сверьте, что в заголовке есть колонки даты и суммы.
- Модуль не заменяет бухгалтерию и не даёт финансовых советов — он структурирует ваши же данные
  (см. `SAFETY.md`: доменные агенты информируют, но направляют к лицензированному финсоветнику).
