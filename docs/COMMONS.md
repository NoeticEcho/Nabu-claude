# Nabu Commons — протокол федеративного самоулучшения

Сеть инстансов Nabu умеет улучшать саму себя: каждый инстанс (с согласия
пользователя) замечает проблемы у себя, предлагает изменения, сеть голосует,
разработчики их реализуют, а **человек-мейнтейнер** ревьюит и мержит. Это
кооперативный, а не автономный процесс: инстансы советуют, решает человек.

> **Выключено по умолчанию.** Функция активна только при `commons.enabled=true`
> в конфиге инстанса. Без явного включения ни одно исходящее действие не выполняется.

## Что это

Цикл самоулучшения сети — восемь шагов, замкнутых в петлю:

```
   ┌──────────────────────────────────────────────────────────────┐
   │                                                              │
   ▼                                                              │
 sense ──▶ propose ──▶ vote ──▶ develop ──▶ review ──▶ human ──▶ release ──▶ measure
(инстанс   (issue     (👍 +   (claim +    (мейн-     merge     (тег,        (инстансы
 замечает  с агре-    evidence  PR)        тейнер:   (только    CHANGELOG)   сверяют
 у себя)   гатами)    комменты)            тесты +   человек)                эффект →
                                           крит.                             новый sense)
                                           поверх.)
   ▲                                                                          │
   └──────────────────────────────────────────────────────────────────────────┘
```

- **sense** — инстанс агрегирует локальные сигналы (частота проблемы, потраченные
  циклы), без личных данных.
- **propose** — создаётся issue по форме `community-proposal` с агрегатами.
- **vote** — сеть выражает приоритет реакциями 👍 и содержательными evidence-комментами.
- **develop** — разработчик заявляет задачу (claim) и открывает PR.
- **review** — мейнтейнер прогоняет тесты и особо смотрит критическую поверхность.
- **human merge** — мерж делает только человек-мейнтейнер.
- **release** — тег + запись в CHANGELOG.
- **measure** — инстансы сверяют, помогло ли изменение, и петля повторяется.

## Роли

- **Инстанс пользователя** (opt-in) — замечает проблемы, формирует предложения и
  голоса. Все исходящие действия — через approval пользователя. Никогда не мержит
  и никогда не вызывает `--approve`.
- **Мейнтейнер** (единственный) — единственная роль, которая ставит `ready-for-dev`
  и выполняет merge. Держит критическую поверхность под особым контролем.
- **Tally-бот** — детерминированный скрипт (`scripts/commons-tally.mjs`), запускается
  еженедельно через `.github/workflows/commons-tally.yml`. Считает приоритеты и
  пишет `docs/COMMUNITY_PRIORITIES.md`. Не принимает решений — только агрегирует.

## Протокол данных

### Метки (labels)

- `community-proposal` — issue является предложением; учитывается tally-ботом.
- `ready-for-dev` — мейнтейнер счёл предложение валидным и готовым к разработке.
- `in-progress` — по задаче есть активный claim (разработчик её взял).

### Машинные блоки

Инстансы обмениваются структурой через HTML-комментарии в телах issue и комментариях.
Они невидимы в рендере GitHub, но парсятся детерминированно. Точные форматы:

Тело предложения (issue body):

```
<!--nabu-proposal version="v0.9.0" category="perf" evidence="true"-->
```

Комментарий с агрегатами (evidence):

```
<!--nabu-evidence version="v0.9.0" signals="triage_multidomain=0.41;false_multidomain=0.63" period_days="30"-->
```

Комментарий-заявка на разработку (claim):

```
<!--nabu-claim version="v0.9.0"-->
```

Метка в PR о связи с предложением:

```
<!--nabu-contribution issue="123" version="v0.9.0"-->
```

Ключи `signals` — пары `имя=значение` через `;`, значения только агрегатные
(доли/счётчики). `period_days` — окно агрегации. Никаких сырых записей.

### Скоринг приоритета (v2)

Детерминированный, пересчитывается еженедельно, только **advisory** — ничего не
решает, лишь подсказывает мейнтейнеру, куда посмотреть.

```
w(evidence)     = 1 + min(1, signals / 10)        # вес по подтверждённости, 1..2
score           = 👍 + 2 × Σ w(по авторам)         # один автор = его MAX-вес коммент
если авторы охватывают ≥2 разных версий nabu:  score += 1   # version diversity
```

Константы (в шапке `scripts/commons-tally.mjs`): `SIGNAL_CAP=10`, `EVIDENCE_MULT=2`,
`DIVERSITY_MIN_VERSIONS=2`, `DIVERSITY_BONUS=1`.

- **Вес по подтверждённости.** `signals` — число агрегатных сигналов (пар
  `name=value` в атрибуте `signals` блока `nabu-evidence`): сколько локальных
  подтверждений проблемы принёс инстанс. Больше сигналов → выше вес, но
  **не линейно**: cap на `SIGNAL_CAP=10` держит вес в диапазоне 1..2, чтобы один
  коммент с сотней сигналов не «надул» приоритет. Голый агрегат (без сигналов) и
  legacy-комменты старого формата весят 1 — **обратная совместимость**.
- **Дедуп по автору.** По каждому автору берётся его коммент с максимальным весом,
  не сумма — анти-спам: серия комментов от одного инстанса не накручивает score.
- **Version diversity.** Если evidence-авторы охватывают ≥2 различных версий nabu,
  `score += 1`: сигнал, воспроизводящийся в разных версиях, ценнее локального.

Evidence по-прежнему весит больше реакции: содержательный агрегат ценнее клика.
Результат tally-бот пишет в `docs/COMMUNITY_PRIORITIES.md` (таблица, отсортировано
по score) и добавляет секцию **«Для мейнтейнера»** — авто-triage:

- 🆕 **новые за 7 дней** — предложения, созданные в окне 7 дней до запуска;
- 🔥 **готовы к `ready-for-dev`** — `score ≥ 5` И ≥2 различных evidence-авторов И
  ещё без метки `ready-for-dev`;
- 💤 **кандидаты на закрытие** — старше 90 дней И `score ≤ 1`.

Triage — **только подсказки**: метки ставит и закрывает issue по-прежнему человек-
мейнтейнер (см. чек-лист ниже). Файл коммитится только при изменении.

## Безопасность (жёстко)

- **Merge — только человек-мейнтейнер.** Ни инстанс, ни бот, ни автоматика не мержат.
- **Инстансы никогда не вызывают `--approve`.** Само-одобрение исключено архитектурно.
- **Все исходящие действия инстанса — через approval пользователя** (веб-кнопка или
  Telegram). Публикация предложения, голос, claim — каждое требует явного согласия.
- **Приватность: только агрегаты.** В issue/комменты уходят счётчики и проценты, не
  записи. Перед отправкой критик-агент скрабит тело (`private`/`vault` не покидают
  инстанс никогда — см. `SAFETY.md`, инвариант 2).
- **Выключено по умолчанию** (`commons.enabled=false`). Opt-in осознанный.
- **Критическая поверхность — особый флаг.** Изменения в hooks, governance/approval,
  vault-крипте и инсталляторах ревьюятся с повышенным вниманием и не проходят
  «на автопилоте», даже с высоким score.

## Sybil-честность

Приоритизация носит **рекомендательный** характер, поэтому накрутка малоосмысленна:

- Накрутка реакций 👍 бессмысленна против **human-merge**: score ничего не мержит,
  он лишь подсказывает мейнтейнеру, куда посмотреть.
- **Evidence-комменты ценнее реакций** именно потому, что содержательны: они несут
  проверяемые агрегаты, а не голый клик. Поэтому вес ×2 — за содержание, не за объём.
- Мейнтейнер валидирует агрегаты глазами перед `ready-for-dev`; выдуманные сигналы
  видны на ревью.

## Команды пользователя

- `/nabu-propose` — сформировать предложение из локальных агрегатов (через approval).
- `/nabu-vote` — проголосовать 👍 или добавить evidence-коммент к предложению.
- `/nabu-contribute` — заявить задачу (claim) и подготовить PR.
- `/nabu-review-pr` — помочь с ревью PR (тесты, критическая поверхность); merge всё равно за человеком.

Детали каждой команды — в `commands/`.

## Для мейнтейнера

Чек-лист обработки предложения:

1. **Валидность** — проблема реальна, агрегаты правдоподобны, категория верна,
   нет личных данных. Невалидное — закрыть с пояснением.
2. **`ready-for-dev`** — поставить метку валидным предложениям; при claim'е появится
   `in-progress`.
3. **Ревью PR** — прогнать тесты (`npm test`, `npm run test:hooks`); **особо** смотреть
   критическую поверхность (hooks / governance / vault / installers); проверить, что
   привязка `<!--nabu-contribution issue=…-->` указывает на валидное предложение.
4. **Merge** — только вручную, только после зелёных тестов и осознанного ревью.
5. **Закрыть issue** — связать с PR/релизом.

Как читать `docs/COMMUNITY_PRIORITIES.md`: список отсортирован по `score` (👍 +
2×evidence). Высокий score — сигнал спроса, **не** мандат. Читайте вместе с
`ROADMAP.md`: инварианты приватности и границы компетенции важнее любого score.
