# Vault (E2E-шифрование приватного контента) — модель и план

## Текущее состояние (v0.8.0)
`visibility` имеет три уровня: `default` · `private` · `vault`. Сейчас **`vault` обрабатывается
как усиленный `private`**:
- эмбеддинги и обработка — только локально (Ollama), никогда в облако/Claude/логи;
- `recall` фильтрует по visibility (vault можно исключить из контекста);
- контент хранится в общей БД как обычный текст (НЕ зашифрован на клиенте).

Это безопасно против утечки в облако/LLM, но **НЕ обеспечивает** обещанного docs/09 §13 /
docs/11 свойства vault: «клиент дешифрует только при открытии; сервер видит лишь шифротекст».

## Целевая модель (E2E, отложено — архитектурно крупное)
Как в основном Nabu (docs/07 §, Tauri Stronghold):
1. Мастер-ключ vault — только на устройстве (Stronghold/локальный keystore), НИКОГДА в БД/облако.
2. Vault-заметки шифруются на клиенте (AEAD, напр. XChaCha20-Poly1305); в БД — только `ciphertext`
   (в `note_versions.ciphertext=true`), сервер/конвейер их не читает.
3. Эмбеддинги vault: считаются локально по расшифрованному тексту, хранятся ЛОКАЛЬНО (SQLite),
   не реплицируются на сервер (как для private в docs/07 §7.3).
4. Конвейер обработки vault запускается ТОЛЬКО в локальном рантайме (Tauri), не на сервере.

## План реализации в Nabu-claude
- `lib`: `VaultCrypto` (шифр/дешифр по локальному ключу), `EmbeddingsRepository` для vault → локальный
  SQLite вместо общей pgvector; `MemoryRepository` пишет `ciphertext` для vault, не plaintext.
- Ключ: читать из локального keystore/`.nabu/state` (не в git — уже в `.gitignore`), не из БД.
- `memory-server`: для visibility=vault — шифровать перед записью, дешифровать при чтении локально;
  recall по vault — только по локальному индексу.
- Тесты: round-trip шифрования; проверка, что в общей БД для vault лежит только шифротекст.

## Пока НЕ сделано — почему
Требует локального keystore-контура и разделения хранилищ (общая pgvector vs локальный SQLite для
vault-эмбеддингов) — это отдельная веха. До неё: **не помечать по-настоящему секретное как `vault`,
ожидая серверного E2E** — использовать `private` (локальная обработка, без клиентского шифрования),
понимая, что контент лежит в общей БД открытым текстом. Инвариант «private/vault не в облако/LLM/логи»
соблюдается уже сейчас.
