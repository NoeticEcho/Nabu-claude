# 0. Назначение и формат

Документ — журнал ключевых архитектурных решений Nabu (Architecture Decision Records). Каждое решение зафиксировано в стандартной форме: контекст (почему встала задача), решение (что выбрано), последствия (положительные и отрицательные), отвергнутые альтернативы (с обоснованием отказа).

ADR — это иммутабельная запись. Если решение пересматривается — создаётся новое ADR со ссылкой Supersedes на предыдущее. Старое получает статус Superseded и сохраняется.

# ADR-001 — NestJS как backend-фреймворк

**Статус: **Accepted**    Дата: **2026-05-20

### Контекст

Серверная часть Nabu координирует десятки агентов, многочисленные хранилища (Postgres, TypeDB, S3, pgvector), очереди и Realtime-канал. Нужна модульная архитектура с явной DI, понятной структурой контроллеров и поддержкой стриминговых ответов.

Рассматривался вариант разместить API внутри Next.js (API-routes). Этот путь привлекателен для маленьких приложений, но не подходит для системы с десятками сервисов, фоновых воркеров и сложным внутренним состоянием.

### Решение

Backend — отдельное NestJS-приложение (apps/api), запускаемое независимо от Next.js. Воркер очередей — то же NestJS-приложение, запущенное с другим entry-point (apps/worker). Scheduler — третий entry-point.

Все агенты Mastra регистрируются как Injectable NestJS-сервисы.

### Положительные последствия

- Явная модульность; каждый функциональный домен — отдельный NestJS-модуль.
- Готовая DI с поддержкой провайдеров, скоупов, middleware, guards, interceptors.
- Поддержка стриминговых ответов и WebSocket из коробки.
- Автогенерация OpenAPI 3.1 спецификации из декораторов контроллеров.
- Стек одинаков для api/worker/scheduler — это один кодовой код с разными entry-point'ами.

### Отрицательные последствия и риски

- Дополнительный процесс (или Docker-сервис) в self-host развёртывании.
- Команда должна знать NestJS-конвенции (Injectable, Module, Controller, decorators).

### Отвергнутые альтернативы

- **Next.js API-routes: **Подходит для простых API, но размывает разделение между UI и серверной логикой. Невозможно элегантно разместить долгие background-jobs в той же кодовой базе. Нет нормальной DI.
- **Fastify напрямую: **Меньше структуры, больше boilerplate'а; нет встроенного DI. Подходящий выбор для микросервисов, не для домена нашего размера.
- **Encore / Nitric: **Молодые фреймворки, экосистема меньше; ниже совместимость с экосистемой NestJS-плагинов и Mastra adapter'а.

# ADR-002 — Mastra.ai интегрируется только через @mastra/nestjs

**Статус: **Accepted**    Дата: **2026-05-20

### Контекст

Mastra.ai по умолчанию запускает собственный Hono-сервер. Это удобно для отдельностоящих агентских приложений, но создаёт два HTTP-сервера в нашем стеке: один — для основной бизнес-логики (NestJS), второй — для агентов (Hono). Управление аутентификацией, rate limiting'ом, observability в двух местах — это лишний источник ошибок.

Команда Mastra выпустила официальный адаптер @mastra/nestjs, который встраивает Mastra прямо в NestJS-приложение: единый HTTP-сервер, единый middleware-стек, DI-совместимая регистрация агентов, поддержка стриминга и MCP transport.

### Решение

Mastra встраивается через @mastra/nestjs. Hono-сервер Mastra НЕ запускается в production. Mastra Studio может использоваться в dev-режиме отдельно для интроспекции и отладки.

Mastra-instance создаётся внутри MastraService.onModuleInit() и регистрирует агенты, инжектированные через DI.

Все endpoint'ы агентов экспонируются через NestJS-контроллеры; raw маршруты Mastra доступны только через адаптер.

### Положительные последствия

- Единая аутентификация — JWT валидируется одним AuthGuard на всё.
- Единый rate limiting через NestJS-middleware.
- Единые structured logs и трейсы с trace_id.
- Graceful shutdown единый — NestJS lifecycle hooks работают для Mastra тоже.
- Поддержка MCP transport нативно.

### Отрицательные последствия и риски

- Привязка к официальному адаптеру: если Mastra в будущем кардинально его переработает — потребуется миграция.
- Некоторые фичи Mastra Studio (live debugger) могут не работать поверх адаптера; для dev требуется отдельная команда mastra studio.

### Отвергнутые альтернативы

- **Отдельный Hono-сервер Mastra: **Дублирование auth/middleware/logs. Сложнее observability.
- **Прямые вызовы Mastra-internals без адаптера: **Хрупкая интеграция, ломается на каждом релизе Mastra.
- **LangGraph вместо Mastra: **Python-стек, ломает unified-TS-стек.

# ADR-003 — Supabase self-hosted как основа backend-слоя

**Статус: **Accepted**    Дата: **2026-05-20

### Контекст

Nabu нуждается в Postgres, аутентификации, реалтайм-канале, объектном хранилище, очередях, расписаниях, edge-функциях. Каждый компонент можно собрать отдельно (Postgres + Hanko + Centrifugo + MinIO + Redis + cron + Cloudflare Workers), но это требует значительного интеграционного усилия.

Supabase агрегирует эти компоненты в единый docker-compose. ВАЖНО: критерий выбора — все нужные компоненты должны работать в self-hosted режиме без vendor lock.

### Решение

Backend-слой — Supabase self-hosted через docker-compose.

Используются: Postgres (с pgvector и pgmq), GoTrue (auth), Realtime, Storage (S3-совместимый Storage API + MinIO как backend), Edge Functions (Deno), pg_cron.

ЗАПРЕЩЕНО использование Supabase-функций, доступных только в Supabase Cloud (например, Supavisor connection pooling — заменяется на PgBouncer; Vault — заменяется на собственный шифровальный слой).

### Положительные последствия

- Один docker-compose разворачивает большинство инфраструктуры.
- Все компоненты open-source и self-hostable.
- Auth-инструмент (GoTrue) умеет email/пароль, magic-link, OAuth, MFA.
- RLS в Postgres даёт декларативную горизонтальную авторизацию.
- Realtime через WebSocket — без отдельной инфраструктуры.

### Отрицательные последствия и риски

- Зависимость от стабильности Supabase open-source релизов. Mitigation: подписка на security-релизы + ежемесячные maintenance updates.
- Сложность настройки backup и обновления в split-deployment.
- Supavisor (cloud-only) недоступен — connection pooling приходится настраивать через PgBouncer вручную.

### Отвергнутые альтернативы

- **Своя сборка компонентов (Postgres + Hanko/Auth.js + Centrifugo + MinIO + Redis + Temporal + cron): **Требует существенно больше работы по интеграции. Каждый компонент — отдельная сетевая конфигурация, отдельная схема логов.
- **PocketBase: **Замечательный для маленьких приложений, не масштабируется под наши потребности (нет pgmq-эквивалента, нет нормального RLS).
- **AWS Amplify / Firebase: **Полный vendor lock. Несовместимо с self-host требованиями.

# ADR-004 — TypeDB 3.0 для графа знаний

**Статус: **Accepted**    Дата: **2026-05-20

### Контекст

Nabu ведёт граф знаний с типизированными сущностями (Person, Project, Goal, Habit, ...) и многоарными отношениями. Требуются: иерархия типов, явные многоарные отношения, правила вывода, читаемые LLM-агентами без перевода в формальную логику.

### Решение

TypeDB 3.0 Community Edition в отдельном Docker-контейнере. Схема (TQL) версионируется как файлы миграций в repo (migrations/typedb/).

TypeDB-driver Node.js используется в NestJS-сервисе TypeDBService. Все операции с графом проходят через этот сервис.

### Положительные последствия

- Native поддержка многоарных отношений (например, decision relates agent, options, criteria, outcome) без эмуляции через джойн-таблицы.
- Иерархия типов: Note исa Document; Source isa Document; Person isa Entity. Запросы по супертипам работают полиморфно.
- Декларативные правила вывода читаются LLM-агентом как утверждения; не нужны императивные транзитивные замыкания в коде.
- Open-source self-hostable; нет vendor lock.

### Отрицательные последствия и риски

- Меньше сообщество и инструментарий, чем у Neo4j.
- TypeDB 3.0 — относительно молодая версия; есть риск breaking changes в minor-релизах.
- Кластеризация только в Enterprise Edition — на self-host остаётся single-node. Для нашего сценария (single-tenant) приемлемо.

### Отвергнутые альтернативы

- **Neo4j Community: **Хорошая зрелость, но только бинарные отношения. Многоарные нужно эмулировать через reification. Cypher не такой хорошо ложится на LLM-промпты как TQL.
- **RDF + SPARQL (Apache Jena, Stardog): **Сильная теоретическая база, но крайне громоздкий синтаксис. LLM-агентам тяжелее формулировать SPARQL-запросы корректно.
- **Graph в Postgres через recursive CTE: **Подходит для простых случаев. Для нашей онтологии с правилами вывода — ловушка: запросы становятся непрозрачными, поддержка тяжёлая.
- **DuckDB / Kuzu: **Embedded graph DBs. Не подходят для серверной многопользовательской работы.

# ADR-005 — Tiptap как редактор Markdown в клиенте

**Статус: **Accepted**    Дата: **2026-05-20

### Контекст

Редактор — центральный элемент UX. Требования: Markdown как канон, WYSIWYG-режим с inline-форматированием, поддержка fenced code, LaTeX, Mermaid, изображений, wikilink-автодополнения, slash-команд, undo/redo, мобильная совместимость.

### Решение

Tiptap (на ProseMirror) как основной редактор. Schema-driven, кастомные ноды для wikilink, embed, slash-menu.

Сохранение в .md происходит через сериализатор Tiptap → Markdown (расширение @tiptap/markdown или собственная реализация).

### Положительные последствия

- ProseMirror-фундамент — индустриальный стандарт сложных редакторов (Notion, Linear, Atlassian тоже на ProseMirror-подобных движках).
- Schema-driven подход даёт строгую валидацию структуры документа.
- Сообщество и плагины большие.
- Хорошо ложится на slash-команды и автодополнение wikilinks.

### Отрицательные последствия и риски

- Сериализация ProseMirror-документа в чистый Markdown — нетривиальна; некоторые сложные конструкции (например, nested blockquotes с code) могут терять fidelity.
- Bundle size больше, чем у CodeMirror-варианта.
- WYSIWYG-режим иногда тяжелее освоить пользователям, привыкшим к raw Markdown (Obsidian). Mitigation: режим переключения «source mode».

### Отвергнутые альтернативы

- **CodeMirror 6: **Идеален для raw Markdown с syntax highlight. Хуже для inline-форматирования и кастомных нод (нужно эмулировать через decorations).
- **Lexical (Meta): **Молодой, активно меняется API. Сейчас рискованно.
- **Slate.js: **Гибкий, но менее зрелый сериализатор Markdown. Сообщество меньше Tiptap.
- **Monaco: **Слишком тяжёлый, ориентирован на код, не на прозу.

# ADR-006 — Tauri 2.x для desktop и mobile

**Статус: **Accepted**    Дата: **2026-05-20

### Контекст

Nabu должна работать на desktop (Windows/macOS/Linux) и mobile (iOS/Android). Требования: единый фронтенд-код, нативные интеграции (file watcher, system tray, hotkeys, share-extension, биометрия), небольшой размер бинаря, low-resource (важно для слабых машин self-host пользователей).

### Решение

Tauri 2.x для desktop и mobile. Та же Next.js SPA-сборка упаковывается в Tauri. Нативные плагины Tauri используются для всех системных интеграций: filesystem, sql (SQLite), store, stronghold, updater, notification, biometric, share-target, deep-link.

Дополнительные нативные слои на iOS/Android реализуются через mobile-плагины Tauri (Swift/Kotlin) — это обеспечивает прохождение App Store Review 4.2.

### Положительные последствия

- Размер бинарника десктоп-приложения 10–50 МБ (vs 100+ МБ у Electron).
- Меньшее потребление памяти (нет встроенного Chromium).
- Tauri Stronghold для безопасного локального хранения ключей (включая vault master key).
- Tauri Updater для самообновления.
- Единая Rust-инфраструктура для системных интеграций.

### Отрицательные последствия и риски

- WebView на разных платформах разный (WebView2 на Win, WKWebView на macOS/iOS, WebKitGTK на Linux). Различия в поддержке CSS/API. Требует регулярного кроссплатформенного тестирования.
- Tauri mobile прямо помечен как «не first-class citizen» — не все desktop-плагины портированы; ряд функций придётся писать на Swift/Kotlin самостоятельно.
- Команда должна знать минимум базовый Rust для нативных плагинов.
- App Store Review 4.2 — без существенного нативного слоя приложение могут отклонить.

### Отвергнутые альтернативы

- **Electron + Capacitor: **Большой размер бинаря, существенное потребление памяти. Capacitor для mobile добавляет ещё один слой.
- **Только нативные SwiftUI/Jetpack Compose + отдельный desktop: **Утроенный объём работы. Несовместимо с timing-ом.
- **Flutter: **Сильный отдельный фреймворк, но не интегрируется с Next.js/React-кодом. Полная переписка.
- **PWA (без обёртки): **Не даёт нативной интеграции, share-extension, биометрии. На iOS PWA остаётся ограниченным.

# ADR-007 — Локальный LLM через Ollama

**Статус: **Accepted**    Дата: **2026-05-20

### Контекст

Категории видимости private и vault требуют, чтобы LLM-обработка происходила локально на машине пользователя, без выхода данных на внешние API.

### Решение

Локальный LLM запускается через Ollama. Nabu обнаруживает Ollama автоматически (стандартный порт 11434) и предлагает использовать.

Mastra обращается к Ollama через стандартный OpenAI-совместимый интерфейс. Никакой специальной интеграции на уровне Nabu не требуется.

Рекомендуемые модели: Qwen 2.5 14B Instruct (общая), Qwen 2.5 Coder 14B (технические задачи), Llama 3.3 70B при достаточных ресурсах GPU.

### Положительные последствия

- Ollama — фактический стандарт для локального запуска LLM на пользовательских машинах. Большое сообщество, активная разработка.
- Нет необходимости интегрировать llama.cpp/exllama на уровне нашего кода.
- Пользователь может менять модель, не меняя Nabu.
- OpenAI-совместимый интерфейс позволяет Mastra работать с Ollama без специального адаптера.

### Отрицательные последствия и риски

- Зависимость от стабильности Ollama. Mitigation: интерфейс — стандартный OpenAI-API, теоретически можно заменить на любой OpenAI-compatible local server (vLLM, LM Studio, llamafile).
- Дополнительный процесс на десктопе пользователя.

### Отвергнутые альтернативы

- **llama.cpp напрямую (как библиотека или binary): **Требует управления моделями и квантизациями на уровне нашего кода. Большая нагрузка на поддержку.
- **LM Studio: **Закрытый код, не подходит для нас.
- **Прямой запуск моделей через Transformers / vLLM: **Тяжело собирать в один бандл с Tauri-приложением. Большой размер.

# ADR-008 — Markdown + YAML frontmatter — канонический формат заметок

**Статус: **Accepted**    Дата: **2026-05-20

### Контекст

Заметки должны быть портируемы между инструментами (Obsidian, Logseq, Tana, обычный текстовый редактор). Пользователь должен иметь возможность забрать свои данные и продолжить работу даже если Nabu перестанет существовать.

### Решение

Канонический формат — Markdown (CommonMark + GFM) с YAML frontmatter в начале файла. Все производные представления (Postgres-метаданные, эмбеддинги, TypeDB-граф) могут быть пересобраны из .md-файлов.

Frontmatter содержит: id, type, title, tags, created, updated, status, domain, visibility, links (массив wikilink'ов), любые domain-specific поля (например, mood для daily-note).

### Положительные последствия

- Полная портируемость: пользователь может работать с .md в любом редакторе.
- Зеркальная совместимость с Obsidian/Logseq vault'ами.
- Простая логика бэкапа и восстановления.
- Diff'ы человекочитаемы; git-friendly.

### Отрицательные последствия и риски

- Некоторые продвинутые конструкции (например, блочные ссылки в стиле Roam — ((block-id))) требуют дополнительной семантики; решается через extension-нотацию в frontmatter.
- Frontmatter может разойтись с фактическим содержимым; нужна валидация (Zod-schema) и автокоррекция.

### Отвергнутые альтернативы

- **Proprietary JSON formats: **Vendor lock, не портируется.
- **OPML / Org-mode: **Org-mode мощный, но узкая аудитория. OPML только для outliner-стиля.
- **Plain text без structure: **Нет места для метаданных.

# ADR-009 — pgvector для эмбеддингов

**Статус: **Accepted**    Дата: **2026-05-20

### Контекст

Семантический поиск и Linker-агент требуют хранения и быстрого top-k поиска эмбеддингов. Объём — несколько эмбеддингов на заметку (контентный + по абзацам); до десятков миллионов векторов для активного пользователя за годы.

### Решение

pgvector внутри основного Postgres-инстанса. Размер вектора по умолчанию — 1024 (под nomic-embed-text). Индекс — HNSW.

Отдельная таблица note_embeddings с колонками (note_id, version_number, model_name, dimension, embedding, content_hash, created_at). PK по (note_id, version_number, model_name).

### Положительные последствия

- Один Postgres-инстанс — одна точка обслуживания, бэкапа, observability.
- Транзакционная согласованность: эмбеддинги обновляются в той же транзакции, что и метаданные.
- RLS работает для эмбеддингов так же, как для остального.
- HNSW в pgvector даёт миллисекундный top-k на корпусах до миллионов векторов.

### Отрицательные последствия и риски

- Postgres не оптимизирован под чисто-векторные рабочие нагрузки на 100M+ векторов. Mitigation: при росте — мигрировать тяжёлые корпусы в выделенный Qdrant, оставив Postgres для метаданных.
- HNSW занимает заметную долю памяти (40-60 байт/вектор). Учитываем в capacity planning.

### Отвергнутые альтернативы

- **Qdrant standalone: **Лучшее purpose-built решение для векторов, но дополнительный сервис в стеке. Излишен для нашего масштаба на старте.
- **Milvus: **Тяжёлый, нагружает infrastructure. Не для self-host single-user.
- **Pinecone: **Cloud-only, vendor lock.
- **LanceDB: **Embedded, не для серверной много-пользовательской работы.

# ADR-010 — Supabase Queues (pgmq) для асинхронных задач

**Статус: **Accepted**    Дата: **2026-05-20

### Контекст

Мультиагентный конвейер — фоновые задачи (агентские вызовы, transcription, OCR, синтез документов, periodic digests). Нужна очередь с guaranteed delivery, exactly-once в окне видимости, ретраи, idempotency.

### Решение

Supabase Queues (pgmq) как основной движок очередей. Воркер на NestJS подписан на pgmq.read с visibility_timeout.

pg_cron — для расписаний (ежедневные дайджесты, ночная переиндексация).

### Положительные последствия

- Никаких дополнительных сервисов (Redis/RabbitMQ/Temporal): очередь — в том же Postgres, что и остальные данные.
- Транзакционная согласованность: вставка задачи и изменение состояния — в одной транзакции (outbox-паттерн).
- Один бэкап покрывает и состояние, и задачи.
- Exactly-once delivery в visibility window.

### Отрицательные последствия и риски

- Пропускная способность pgmq ограничена производительностью Postgres. До ~10K msg/s — нормально; выше — нужны выделенные queue-системы.
- Нет встроенных приоритетов (workaround — отдельные очереди для приоритетных задач).
- Меньше готовых dashboards/observability tools по сравнению с Temporal/BullMQ.

### Отвергнутые альтернативы

- **Redis + BullMQ: **Дополнительный сервис; не транзакционно с Postgres; нужна отдельная стратегия бэкапа Redis.
- **Temporal: **Мощный workflow-engine, но overkill для нас. Большая операционная сложность.
- **RabbitMQ / NATS: **Дополнительный сервис; не привязан к нашей основной БД.

# ADR-011 — CopilotKit как клиентский слой генеративного UI

**Статус: **Accepted**    Дата: **2026-05-20

### Контекст

Nabu — мультиагентная среда, где UI должен динамически перестраиваться под текущий контекст и команды пользователя. Нужен фреймворк для inline chat, slash-команд, генеративных компонентов, передачи состояния frontend↔agents.

# ADR-012 — Три категории видимости с разной LLM-маршрутизацией

**Статус: **Accepted**    Дата: **2026-05-20

### Контекст

Пользовательские данные неоднородны по чувствительности: исследования и проекты — терпят внешние LLM-API; КПТ-дневник — не должен покидать локальную машину; медицинские/юридические записи — даже сервер не должен иметь доступ к их содержимому.

### Решение

Три категории видимости: default, private, vault.

default: обработка через внешний LLM-провайдер (Anthropic API). Сервер видит контент.

private: обработка только через локальный Ollama. Содержимое НЕ уходит на внешние API. Сервер хранит содержимое (зашифрованным в покое, но не E2E).

vault: содержимое шифруется на клиенте E2E (envelope encryption, AES-256-GCM + AES-256-KW, ключ дериверован Argon2id из пользовательского пароля). Сервер хранит только шифротекст; LLM-обработка только на локальном Ollama и только из desktop-клиента.

Категория назначается пользователем явно или Domain Classifier (с подтверждением для чувствительных). Изменение категории — аудируемое событие.

### Положительные последствия

- Чёткий контракт приватности, легко объяснимый пользователю.
- Технически разделённые маршруты — невозможно «случайно» отправить private-контент на внешний API (тесты автоматически проверяют это).
- Vault даёт техническую защиту от компрометации сервера.

### Отрицательные последствия и риски

- vault-данные становятся недоступны в веб-клиенте (нет пароля для расшифровки в браузере без UI-формы; реализация возможна, но компромисс выбран в пользу desktop-only).
- Утеря vault-пароля = безвозвратная потеря содержимого; UX обязан явно об этом предупреждать.
- Сложность тестирования (тесты должны проверять маршрутизацию).

### Отвергнутые альтернативы

- **Только двоичная public/private: **Слишком грубо: либо «всё видно», либо «всё E2E». Не покрывает реальный спектр чувствительности.
- **Шифрование всего E2E: **Делает невозможной серверную агентскую обработку для большинства данных. Сильно ограничивает ценность Nabu.
- **Доверие cloud-провайдеру: **Не приемлемо для психотерапевтических данных.

# ADR-013 — Sync engine развивается поэтапно

**Статус: **Accepted**    Дата: **2026-05-20

### Контекст

Многоплатформенный клиент требует синхронизации. Сложность конфликт-разрешения существенно растёт от last-write-wins к CRDT. Реализовать сразу полноценный CRDT — большая инвестиция; нужно поэтапно.

### Решение

Ph6: Sync engine v1 — last-write-wins по hash + updated_at. «Проигравшая» версия НЕ удаляется, сохраняется как conflict-suspended.

Ph7: Sync engine v2 — 3-way merge для текстовых конфликтов с использованием base-версии (последняя синхронизированная).

Ph7 (опционально): CRDT (Yjs или Automerge) для файлов, помеченных как collaborative. Реалтайм-presence через Supabase Realtime.

### Положительные последствия

- Каждый этап даёт работоспособный результат.
- LWW покрывает 90%+ пользовательских конфликтов (одно устройство правит и забывает; второе подхватывает).
- 3-way merge решает большинство кейсов «обе сделали правки разных абзацев».
- CRDT — только для активной коллаборации, узкий use-case.

### Отрицательные последствия и риски

- Пользователи, нуждающиеся в realtime-collaboration с первого дня, придётся ждать Ph7.
- Миграция данных между этапами требует осторожности (особенно при переходе к CRDT для уже существующих файлов).

### Отвергнутые альтернативы

- **CRDT с первого этапа: **Большой объём работы; CRDT-файлы тяжелее в сериализации к чистому Markdown.
- **Серверный авторитет без локальной правки: **Несовместимо с local-first принципом.

# ADR-014 — bun как runtime и менеджер пакетов

**Статус: **Accepted**    Дата: **2026-05-20

### Контекст

TypeScript-стек. Стандартный путь — Node.js + npm/pnpm. Альтернативы — bun или Deno.

### Решение

bun используется в качестве runtime для NestJS-сервера и worker'а, и как менеджер пакетов в монорепо.

Node.js compatibility-mode bun'а покрывает потребности NestJS + Mastra.

### Положительные последствия

- Существенное ускорение бутстрапа и тестов (3-5x на типовых проектах).
- Встроенный менеджер пакетов, тестовый раннер, бандлер — меньше отдельных tools.
- Хорошая обратная совместимость с Node.js.

### Отрицательные последствия и риски

- Bun — относительно молодой runtime. Некоторые edge-case'ы Node.js могут вести себя иначе. Mitigation: smoke-тесты на CI с обоими (Node.js fallback).
- Часть npm-пакетов с native-зависимостями может требовать дополнительной настройки.

### Отвергнутые альтернативы

- **Node.js LTS: **Стабильность, но медленнее; больше отдельных tools (pnpm + tsx + vitest).
- **Deno: **Хороший runtime, но другая экосистема; меньше совместимости с NestJS.

# ADR-015 — Argon2id для парольных хэшей и vault-ключа

**Статус: **Accepted**    Дата: **2026-05-20

### Контекст

Нужна стойкая функция деривации ключа из пароля. Применяется в двух местах: хэширование auth-паролей; деривация vault master_key из vault-пароля.

### Решение

Argon2id (мемори-зависимая KDF, рекомендована OWASP) для обоих случаев.

Параметры: memory=64 MB, iterations=3, parallelism=4. Соответствуют OWASP recommendation 2023+ для interactive logins.

Соль — 16 случайных байт на пользователя; хранится в auth.users.encrypted_password (для GoTrue — автоматически) и в user_vault.salt (для vault — явно).

### Положительные последствия

- Защита от GPU-brute-force (благодаря memory-hardness).
- Стандартный выбор в индустрии после провала bcrypt в условиях современных GPU.
- Доступно в Node/Bun через @node-rs/argon2 или argon2-browser для клиента.

### Отрицательные последствия и риски

- Memory-cost замедляет логин (300-500 мс); приемлемо.
- Vault-операции на десктопе с дешёвой батареей могут быть заметными по времени; пользователь видит спиннер.

### Отвергнутые альтернативы

- **bcrypt: **Не покрывает GPU-stress адекватно. OWASP больше не рекомендует.
- **scrypt: **Подходит, но Argon2id — победитель Password Hashing Competition и предпочтителен.
- **PBKDF2: **Слабее против GPU-атак.

# ADR-016 — Event-driven архитектура между слоями

**Статус: **Accepted**    Дата: **2026-05-20

### Контекст

Между слоями Multi-Agent Pipeline есть выбор: прямые синхронные вызовы либо асинхронные события через очередь.

### Решение

Все межслоевые взаимодействия — через события в pgmq. Внутрислоевые — обычные NestJS service-вызовы.

Каждое событие имеет тип, trace_id, user_id, payload. Подписчики — NestJS-сервисы соответствующего слоя.

### Положительные последствия

- Горизонтальная масштабируемость воркеров без перепроектирования.
- Декаплинг: смена реализации одного слоя не ломает остальные.
- Трассируемость и replayability: можно перевыполнить обработку события из лога.
- Естественная гранулярность для retry и idempotency.

### Отрицательные последствия и риски

- Латентность: каждый шаг — отдельный SQL-call. Mitigation: внутри одного шага агенты вызываются синхронно параллельно.
- Локальная отладка чуть сложнее (нужно смотреть трейс через несколько событий).

### Отвергнутые альтернативы

- **Прямые вызовы внутри monolith-сервиса: **Тесная связность, плохо масштабируется.
- **Reactive (RxJS) внутри одного процесса: **Не даёт устойчивости к падению процесса.

# ADR-017 — Промпты — как код в репозитории

**Статус: **Accepted**    Дата: **2026-05-20

### Контекст

Промпты — ключевой фактор качества агентов. Альтернативы хранения: в БД (для горячих обновлений без релиза), в файлах репозитория, в специализированной prompt-management системе.

### Решение

Промпты — .md-файлы в packages/prompts/<agent>/vN.md с YAML-frontmatter (model, vars, eval_ref). Версионируются Git'ом. Активная версия указывается в config.toml.

При деплое инстанс прибит к конкретному коммиту, и поэтому к конкретным промптам.

Изменение промпта = PR с автоматическим eval-run.

### Положительные последствия

- Полная воспроизводимость: pinned версия промпта в коммите.
- Code review для промптов наравне с кодом.
- Eval-suite запускается на каждом PR; регрессии видны сразу.
- Diff'ы промптов в git-blame.

### Отрицательные последствия и риски

- Горячее обновление невозможно без деплоя. Mitigation: rolling deploy быстрый; критические правки делаются hotfix-релизом.
- Нужен disciplined eval-suite иначе теряется польза версионирования.

### Отвергнутые альтернативы

- **Промпты в БД: **Размывает источник истины: production-инстанс может расходиться с тем, что в репо.
- **Сторонние prompt-managers (LangSmith, Helicone): **Vendor lock, дополнительная инфраструктура.

# ADR-018 — Монорепозиторий с workspaces

**Статус: **Accepted**    Дата: **2026-05-20

### Контекст

Структура кода: монорепо (одно репо со всеми приложениями и пакетами) или polyrepo (отдельные репо).

### Решение

Монорепо с bun workspaces (или pnpm workspaces при fallback). Структура apps/* и packages/* (см. SAD §9.1).

### Положительные последствия

- Atomic коммиты, затрагивающие фронт и бэк.
- Shared types (DTO, events) в одном пакете, не дублируются.
- Один CI-pipeline на все приложения.
- Проще onboarding нового разработчика (включая AI-агента: один clone и весь контекст).

### Отрицательные последствия и риски

- Растущий размер репо при долгой истории.
- Необходимость дисциплины — не зависеть apps/* друг от друга напрямую.

### Отвергнутые альтернативы

- **Polyrepo: **Каждое приложение — отдельное репо. Хуже для shared types, для atomic-изменений; больше CI-инфраструктуры.
