# 0. Назначение и формат документа

Документ содержит decomposition объёма работ Nabu в виде Agile-backlog'а. Декомпозиция трёхуровневая: эпик → фича → user story. Документ оптимизирован под автоматическое исполнение AI-coding командой через оркестратор Aperant.

## 0.1. Соглашения по идентификаторам

- E-NN — эпик (10 эпиков: 8 фазовых Ph0…Ph7 плюс 2 cross-cutting).
- F-NN.NN — фича внутри эпика.
- US-NN.NN.NN — user story внутри фичи (story-номер локален для фичи).
- FR-XXXX / NFR-XXXX — требования из SRS (Nabu · SRS).

## 0.2. Поля каждой user story

- Заголовок: ID + краткое название.
- Эпик · Фича · Этап — где живёт история.
- SP (story points) — оценка по Fibonacci (1, 2, 3, 5, 8, 13). Эпики с историей >13 SP подлежат дальнейшей декомпозиции.
- Pri — приоритет MoSCoW: M (Must), S (Should), C (Could).
- Связь с SRS — список FR/NFR из спецификации (трассируемость).
- Зависимости — другие user stories, которые должны быть выполнены первыми.
- Тело — стандартный шаблон «Как [роль], я хочу [возможность], чтобы [польза]».
- Критерии приёмки — список Given-When-Then утверждений.
- Вне scope — что НЕ входит (защита от расширения объёма AI-агентом).
- Технические указания — конкретные подсказки исполнителю (библиотеки, паттерны, ограничения).

## 0.3. Дисциплина исполнения для AI-команды

1. Каждая история берётся в работу только при выполненных зависимостях и подтверждённом готовом коде из предыдущих историй.
1. Перед исполнением AI-агент обязан прочитать связанные FR/NFR из SRS.
1. Исполнение происходит в изолированной git worktree (нативно поддерживается Aperant).
1. CI на PR обязателен: lint, type-check, unit, integration, eval (для агентов). PR не сливается без прохождения.
1. Human review обязателен для: всего, что касается психотерапевтических агентов, безопасности, биллинга, удаления данных, изменения политик приватности.
1. Critic-агент сервера НЕ заменяет human review: это разные слои защиты.

# E-01 — Foundation

**Этап: **`Ph0`

**Цель эпика: **Рабочая инфраструктура без ИИ: аутентификация, базовая модель данных, CRUD заметок в Markdown, минимальный UI. После эпика продукт уже полезен как простой Markdown-вакуум: можно регистрироваться, импортировать корпус, редактировать заметки, экспортировать обратно.

**Definition of Done эпика:**

- Self-host инстанс разворачивается одной командой docker-compose up на чистом VPS с Ubuntu 22.04+.
- Зарегистрированный пользователь за ≤ 5 минут проходит сценарий: регистрация → импорт MD-vault → создание новой заметки → редактирование → экспорт.
- RLS-политики Postgres покрывают все таблицы и проверены автоматическими unit-тестами (попытка чтения чужих данных приводит к 0 строкам, попытка записи — к ошибке).
- Полный экспорт данных в zip генерируется ≤ 60 секунд для корпуса 5 000 заметок.
- Документация: README по развёртыванию, OpenAPI 3.1 спецификация API, ADR по выбору стека.
- Покрытие unit-тестами серверного кода ≥ 70%.

### F-01.01 — Развёртывание Supabase self-hosted

`US-01.01.01 · `**Поднять Supabase через docker-compose**

**Эпик · Фича · Этап: **`E-01 · F-01.01 · Ph0`

**SP · Pri: **`3 · M`

**Связь с SRS (FR/NFR): **`NFR-7002, NFR-7006, NFR-7007`

**Зависимости: **`—`

Как администратор self-host инстанса, я хочу развернуть Supabase одной командой, чтобы получить рабочий backend-стек без ручной настройки.

**Критерии приёмки:**

- **Дано **чистая VM с Ubuntu 22.04 и установленным Docker; **когда **выполняется docker-compose up в директории infra/supabase; **тогда **стартуют контейнеры postgres, gotrue, postgrest, realtime, storage, kong (api gateway), studio, edge runtime; все health-check'и зелёные.
- **Дано **.env заполнен по примеру .env.example; **когда **запрашиваются эндпоинты /auth/v1/health, /rest/v1/, /realtime/v1/; **тогда **все возвращают 200 OK.
- **Дано **pgmq и pgvector установлены через init-scripts; **когда **выполняется SELECT * FROM pg_extension WHERE extname IN ('pgmq', 'vector'); **тогда **обе строки присутствуют.
**Вне scope этой истории:**

- Production hardening (HTTPS, reverse proxy, backup) — отдельная история US-01.01.02
- Кастомные SMTP/SMS провайдеры — отдельные истории
**Технические указания:**

- Использовать официальный supabase/docker репозиторий как основу.
- В default docker-compose.yml включить pgvector и pgmq extensions через init-scripts (см. supabase/.../init-scripts).
- Сгенерировать ключи через generate-keys.sh, никогда не коммитить реальные ключи.
`US-01.01.02 · `**Production hardening развёртывания**

**Эпик · Фича · Этап: **`E-01 · F-01.01 · Ph0`

**SP · Pri: **`5 · M`

**Связь с SRS (FR/NFR): **`NFR-3001, NFR-8004`

**Зависимости: **`US-01.01.01`

Как администратор, я хочу HTTPS, reverse proxy и регулярный бэкап, чтобы инстанс был готов к реальной работе.

**Критерии приёмки:**

- **Дано **развёрнут Caddy (или Nginx) как reverse proxy перед kong; **когда **клиент идёт по https://supabase.example.com/auth/v1/health; **тогда **получает 200 OK с валидным TLS 1.3 сертификатом от Let's Encrypt.
- **Дано **настроен pg_dump cron-задачей каждые 24 часа; **когда **проходит сутки; **тогда **в S3-бэкап-bucket появляется новый дамп с retention 30 дней.
- **Дано **MinIO bucket для бэкапов настроен на отдельном диске или внешнем S3; **когда **выполняется ручная проверка восстановления из дампа на staging; **тогда **восстановление проходит без ошибок.
**Технические указания:**

- Использовать Caddy для простоты — он сам управляет Let's Encrypt сертификатами.
- Бэкап дампа TypeDB — через typedb export команду в отдельном cron-задании.
- Не использовать дефолтные пароли из .env.example в production.

### F-01.02 — Базовая модель данных и миграции

`US-01.02.01 · `**Схема Postgres: пользователи, заметки, версии**

**Эпик · Фича · Этап: **`E-01 · F-01.02 · Ph0`

**SP · Pri: **`5 · M`

**Связь с SRS (FR/NFR): **`FR-2002, FR-2003, FR-2004, NFR-8001, NFR-8002`

**Зависимости: **`US-01.01.01`

Как backend-разработчик, я хочу создать миграции для базовой схемы, чтобы хранить пользователей и их заметки с историей.

**Критерии приёмки:**

- **Дано **применены миграции 001_init_schema.sql; **когда **проверяется наличие таблиц; **тогда **присутствуют: users (link to auth.users), notes, note_versions, tags, tag_links, audit_log.
- **Дано **вставлены тестовые данные двух пользователей через скрипт seed.sql; **когда **пользователь A читает SELECT * FROM notes под своим JWT; **тогда **получает только notes.user_id = A; попытка чтения чужих возвращает 0 строк.
- **Дано **попытка UPDATE notes WHERE user_id = B под JWT пользователя A; **когда **выполняется через PostgREST; **тогда **возвращает 0 affected (RLS блокирует).
**Технические указания:**

- Все таблицы — owner = postgres, USAGE granted to authenticated and service_role.
- RLS policy для notes: USING (user_id = auth.uid()) WITH CHECK (user_id = auth.uid()).
- Колонки note_versions: id (uuid pk), note_id (fk), version_number (int), content_hash (text), s3_key (text), created_at.
- Колонки notes frontmatter — JSONB колонка для произвольных метаданных, плюс explicit колонки для часто-используемых (title, type, status, visibility).
`US-01.02.02 · `**Включить pgvector и создать таблицу эмбеддингов**

**Эпик · Фича · Этап: **`E-01 · F-01.02 · Ph0`

**SP · Pri: **`2 · M`

**Связь с SRS (FR/NFR): **`NFR-8006`

**Зависимости: **`US-01.02.01`

Как backend-разработчик, я хочу хранить эмбеддинги заметок для семантического поиска, чтобы поддерживать FR-1003 (семантический поиск) на этапе Ph1.

**Критерии приёмки:**

- **Дано **pgvector extension установлен; **когда **выполняется CREATE TABLE note_embeddings; **тогда **таблица создаётся с колонкой embedding vector(1024) и индексом ivfflat или hnsw.
- **Дано **вставлена 1000 тестовых векторов через скрипт; **когда **выполняется SELECT по cosine distance с лимитом 10; **тогда **запрос отвечает за ≤ 100 мс.
**Технические указания:**

- Размер вектора зависит от используемой embedding модели — взять 1024 как default (nomic-embed-text); конфигурируется.
- Колонки: note_id (fk), version_number (int), model_name (text), dimension (int), embedding (vector), content_hash (text), created_at. PK по (note_id, version_number, model_name).
- Индекс HNSW предпочтителен (быстрее на чтение, чем IVF Flat) для размеров корпуса до 1M.

### F-01.03 — Аутентификация и управление аккаунтом

`US-01.03.01 · `**Регистрация по email+пароль**

**Эпик · Фича · Этап: **`E-01 · F-01.03 · Ph0`

**SP · Pri: **`3 · M`

**Связь с SRS (FR/NFR): **`FR-1001, NFR-3002, NFR-3008`

**Зависимости: **`US-01.01.01`

Как новый пользователь, я хочу создать аккаунт через email и пароль, чтобы начать пользоваться Nabu.

**Критерии приёмки:**

- **Дано **пользователь вводит валидный email и пароль ≥ 12 символов; **когда **нажимает Sign Up; **тогда **GoTrue создаёт запись, отправляет письмо подтверждения, пользователь видит экран «проверьте почту».
- **Дано **пользователь вводит пароль < 12 символов или из словаря top-10000; **когда **нажимает Sign Up; **тогда **форма показывает ошибку, GoTrue не вызывается.
- **Дано **6 попыток регистрации с одного IP за 1 минуту; **когда **выполняется 7-я; **тогда **rate limit возвращает 429 Too Many Requests.
**Технические указания:**

- Использовать встроенный GoTrue, не строить свою auth.
- Парольная политика — на стороне frontend (UX) и backend (зашита в edge function check_password_policy).
- Rate limit — через Kong (gateway) или middleware в NestJS, но не на стороне frontend.
`US-01.03.02 · `**Вход через email+пароль и magic-link**

**Эпик · Фича · Этап: **`E-01 · F-01.03 · Ph0`

**SP · Pri: **`3 · M`

**Связь с SRS (FR/NFR): **`FR-1002, FR-1003`

**Зависимости: **`US-01.03.01`

Как пользователь, я хочу входить в систему по паролю или magic-link, чтобы иметь удобный и безопасный способ возврата в Nabu.

**Критерии приёмки:**

- **Дано **верные email и пароль; **когда **клиент вызывает /auth/v1/token?grant_type=password; **тогда **возвращается JWT + refresh-token, пользователь попадает в Inbox.
- **Дано **5 неверных попыток подряд; **когда **выполняется 6-я; **тогда **аккаунт временно блокируется на 15 минут, пользователь видит соответствующее сообщение.
- **Дано **пользователь запрашивает magic-link; **когда **получает письмо и кликает по ссылке; **тогда **сессия открывается без ввода пароля.
`US-01.03.03 · `**Удаление аккаунта с полным purge**

**Эпик · Фича · Этап: **`E-01 · F-01.03 · Ph0`

**SP · Pri: **`5 · M`

**Связь с SRS (FR/NFR): **`FR-1007, NFR-9001, NFR-9006`

**Зависимости: **`US-01.02.01, US-01.04.02`

Как пользователь, я хочу полностью удалить свой аккаунт и все данные, чтобы реализовать своё право на удаление (RTBF).

**Критерии приёмки:**

- **Дано **пользователь нажимает Delete account и подтверждает паролем; **когда **стартует процесс удаления; **тогда **пользователь видит экран «удаление запланировано, 24 часа на отмену».
- **Дано **прошло 24 часа без отмены; **когда **запускается ночная задача purge_user; **тогда **удаляются строки во всех таблицах с user_id, объекты в S3 (через MinIO bulk delete), записи в pgvector, related в TypeDB; user в auth.users остаётся как tombstone на 7 дней для аудита, затем удаляется.
- **Дано **прошло ≤ 24 часов; **когда **пользователь нажимает Cancel deletion в письме; **тогда **процесс отменяется, данные не трогаются.
**Технические указания:**

- Удаление S3-объектов: использовать prefix deletion через MinIO API.
- Удаление в TypeDB: написать TQL-запрос match $x has user_id = '...'; delete $x;.
- Удаление в бэкапах: отдельная задача в течение 90 дней (см. NFR-9006).

### F-01.04 — CRUD заметок

`US-01.04.01 · `**Создание заметки**

**Эпик · Фича · Этап: **`E-01 · F-01.04 · Ph0`

**SP · Pri: **`3 · M`

**Связь с SRS (FR/NFR): **`FR-2001, FR-2002, FR-2003, FR-2016`

**Зависимости: **`US-01.02.01`

Как пользователь, я хочу создать новую заметку в Nabu, чтобы записать мысль или фрагмент знания.

**Критерии приёмки:**

- **Дано **пользователь нажимает «новая заметка» и вводит контент в редакторе; **когда **нажимает «сохранить» (или автосохранение через 5 секунд бездействия); **тогда **контент пишется в S3 как .md с YAML frontmatter; в Postgres создаётся строка notes; возвращается id заметки.
- **Дано **заметка содержит `mermaid и `python блоки + LaTeX $E=mc^2$; **когда **сохраняется и переоткрывается; **тогда **контент отображается без потерь, fence-блоки парсятся корректно.
- **Дано **автосохранение в течение 30 секунд после последнего ввода; **когда **ничего не происходит больше; **тогда **заметка сохранена; индикатор «сохранено N сек назад» отображается.
**Технические указания:**

- Редактор: CodeMirror 6 (легче) или Tiptap (богаче UX). Решение зафиксировано в ADR-005.
- S3 key: notes/{user_id}/{note_id}/{version_number}.md.
- Параллельная запись в S3 и Postgres — координируется через NestJS-сервис в одной транзакции.
`US-01.04.02 · `**История версий и откат**

**Эпик · Фича · Этап: **`E-01 · F-01.04 · Ph0`

**SP · Pri: **`5 · M`

**Связь с SRS (FR/NFR): **`FR-2004, FR-2005`

**Зависимости: **`US-01.04.01`

Как пользователь, я хочу видеть прошлые версии заметки и откатывать к ним, чтобы не терять данные при ошибочных правках.

**Критерии приёмки:**

- **Дано **в заметке сделано 5 правок с интервалом; **когда **пользователь открывает «версии»; **тогда **видит список из 6 версий (начальная + 5) с timestamp.
- **Дано **пользователь выбирает версию 2 и нажимает Restore; **когда **система спрашивает подтверждение; **тогда **после подтверждения создаётся новая версия (текущая+1), содержимое = версия 2; «откаченная» версия НЕ удаляется.
- **Дано **размер заметки > 1 МБ; **когда **выполняется сравнение версий; **тогда **diff показывается без падения интерфейса (виртуализированный рендер).
**Технические указания:**

- Хранить каждую версию отдельным S3-объектом, не пытаться использовать только S3 Object Versioning (нужны метаданные).
- Diff на frontend через diff-match-patch или jsdiff.

### F-01.05 — Импорт MD-корпусов

`US-01.05.01 · `**Импорт одного .md файла или директории**

**Эпик · Фича · Этап: **`E-01 · F-01.05 · Ph0`

**SP · Pri: **`5 · M`

**Связь с SRS (FR/NFR): **`FR-2011, FR-2016`

**Зависимости: **`US-01.04.01`

Как пользователь, я хочу загрузить существующий MD-файл или директорию, чтобы перенести свой корпус в Nabu без потерь.

**Критерии приёмки:**

- **Дано **пользователь перетаскивает в drop-zone директорию из 100 файлов; **когда **начинается импорт; **тогда **каждый файл сохраняется как отдельная заметка; frontmatter парсится и кладётся в notes-колонки; теги (#tag) извлекаются и кладутся в tags/tag_links.
- **Дано **в файле есть wikilinks [[Other Note]]; **когда **идёт импорт; **тогда **ссылки сохраняются как есть в .md; в Postgres создаётся pending запись в links для разрешения после полного импорта.
- **Дано **в файле есть relative-пути к картинкам (./image.png); **когда **идёт импорт; **тогда **если соответствующий файл найден в импортируемой структуре, он загружается в S3 и относительные пути в .md заменяются на абсолютные s3:// URI.
**Вне scope этой истории:**

- Импорт из Obsidian-vault с сохранением всех плагин-данных (US-03.02.01 на Ph2)
- Импорт из Notion-export

### F-01.07 — Базовый веб-клиент

`US-01.07.01 · `**Inbox: список последних заметок**

**Эпик · Фича · Этап: **`E-01 · F-01.07 · Ph0`

**Критерии приёмки:**

**Технические указания:**

- Маршрутизация определяется в Mastra-router-конфиге: для visibility=private — провайдер ollama_local.

`US-05.02.01 · `**Vault — E2E шифрование**

**Эпик · Фича · Этап: **`E-05 · F-05.02 · Ph4`

**SP · Pri: **`13 · M`

**Связь с SRS (FR/NFR): **`FR-4019, NFR-3006, NFR-9004`

**Зависимости: **`US-05.01.01`

Как пользователь, я хочу шифровать самые чувствительные заметки локальным ключом, чтобы даже компрометация моего серверного инстанса не раскроет их содержимое.

**Критерии приёмки:**

- **Дано **пользователь выбирает visibility=vault при создании заметки; **когда **клиент шифрует содержимое локально (AES-256-GCM с ключом, дериверованным из пароля через Argon2id) перед отправкой; **тогда **в S3 и Postgres хранится только шифротекст; сервер не имеет ключа.
- **Дано **сервер пытается обработать vault-заметку обычным конвейером; **когда **запускается обработка; **тогда **конвейер ничего не может сделать кроме хранения; пользователь видит «vault-заметки обрабатываются только локально».
- **Дано **пользователь сменил пароль; **когда **выполняется; **тогда **клиент перешифровывает все vault-заметки новым ключом и загружает; до завершения процесса заметки помечены как rekeying.
**Технические указания:**

- Argon2id с параметрами memory=64MB, iterations=3, parallelism=4.
- Ключ НЕ хранится на сервере НИКАК. Если пользователь его забыл — vault потерян (явное предупреждение в UX).
- Соль хранится в metadata пользователя (это не секрет, но должна быть стабильной).
- На каждый vault-объект — отдельный data key, зашифрованный мастер-ключом (envelope encryption) — позволяет ротировать без перешифровки контента.
`US-05.03.01 · `**CBT Agent для распознавания искажений**

**Эпик · Фича · Этап: **`E-05 · F-05.03 · Ph4`

**Связь с SRS (FR/NFR): **`FR-4010, FR-4015`

**Критерии приёмки:**

- **Дано **заметка-журнал по схеме ABC с дисфункциональной мыслью «Если я не справлюсь с этой задачей — я бесполезен»; **когда **запускается CBT Agent; **тогда **идентифицируется минимум 2 искажения (all-or-nothing, labeling); предлагается реструктуризация в Сократовом ключе (вопросы, а не утверждения).
- **Дано **agent выдаёт ответ; **когда **пользователь читает; **тогда **в конце ответа явное disclaimer: «Я — не клиническая интервенция. При тяжёлых состояниях обратитесь к специалисту».

# Приложение Б. Метаправила для AI-исполнителя

1. Никогда не расширяй scope user story. Если требуется больше — оформи новую историю и зависимость, не правь текущую.
1. Если требование из SRS противоречит acceptance criteria — приоритет у SRS. Подними противоречие в комментарии PR, не решай в одностороннем порядке.
1. Если зависимая user story не выполнена — не бери задачу. Aperant должен ждать.
1. Если acceptance criteria недостижимы из-за внешних причин (флакающая зависимость, недоступный API) — оформи это явно, не маскируй mock'ами в проде.
1. Всегда добавляй автоматические тесты на новые публичные API.
1. Промпты агентов — это код. Версионируй, рецензируй PR, запускай eval.
1. Не вводи проприетарные форматы. Канон — Markdown + YAML frontmatter.
1. Не используй облачные функции Supabase, отсутствующие в self-hosted сборке.
1. При любом вмешательстве в категории видимости (private/vault) — обязательное предупреждение в PR, обязательный human review.
1. Принцип «no surprises»: пользователь должен иметь возможность объяснить любое поведение системы. Каждый агент — observable, каждый вывод — explainable.
