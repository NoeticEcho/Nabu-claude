# OlimpOS — руководство пользователя

OlimpOS — много-тенантный режим Nabu: личный ИИ-«Совет» превращается в совместную платформу, где
несколько пользователей ведут личные и командные проекты, программируют в песочницах, публикуют сайты
и делятся агентами. Это руководство — для конечных пользователей и владельца инстанса.

> **Развёртывание сервера** (HTTPS, reverse-proxy, env) описано отдельно в [`DEPLOY_HTTPS.md`](DEPLOY_HTTPS.md).
> Архитектура и фазы — в [`OLIMPOS_ROADMAP.md`](OLIMPOS_ROADMAP.md).

---

## 1. Как устроена изоляция (важно понимать сразу)

- **Личное пространство** `u:<ваш-id>` — ваша память, знания, задачи, финансы, здоровье. Оно
  **никогда** не смешивается с данными других пользователей. Это жёсткий инвариант платформы.
- **Пространство проекта/группы** `g:<chat>` — общая память и задачи команды над одним проектом.
  Реплики в групповом чате атрибутируются по автору, но живут в пространстве проекта, а не в личном.
- **Общий слой** `__commons__` — сюда попадают только опубликованные агенты/скиллы/процедуры и
  обезличенный опыт. Личная память в общий слой не уходит.

Медицина, финансы и отношения по умолчанию `private` и не отправляются во внешние сервисы.

## 2. Регистрация и вход

Есть два пути — выбирайте любой, аккаунт один и тот же.

### Через Telegram (быстрее всего)
1. Напишите боту команду `/start`. Бот заведёт ваше личное пространство автоматически.
2. Для входа в веб-интерфейс: на странице логина нажмите **«Войти через Telegram»** — откроется
   бот со ссылкой `t.me/<bot>?start=…`; подтвердите, и веб-сессия создастся сама (код одноразовый, 10 мин).

### Через email + пароль
1. На странице логина переключитесь на **«Регистрация»**, введите email и пароль (≥ 8 символов).
2. Далее входите этой парой. Пароль хранится в виде scrypt-хеша, в открытом виде — нигде.

Выход — кнопка logout; сессия живёт 30 дней (`HttpOnly`-cookie).

## 3. Общение с адъютантом

Как и в одно-пользовательском Nabu: пишете адъютанту в веб-чат или в личку боту. На сложные
много-доменные запросы созывается Совет экспертов, решение синтезируется с честными trade-off’ами.
Вся ваша память доступна Совету — но только **ваша**.

## 4. Проекты и agile-доска

Открывается кнопкой **🛠 OlimpOS** в шапке веб-чата. Три вкладки:

### Проекты
- Создать проект: название + (необязательно) цель → **Создать**.
- У каждого проекта есть своя доска — кнопка **«Доска →»**.

### Доска (Kanban)
Четыре колонки: **К выполнению → В работе → Ревью → Готово**.
- Добавить задачу — поле сверху (попадает в «К выполнению»).
- Перемещать задачу — кнопки на карточке (в любую другую колонку).
- **+ Эпик**, **+ Спринт** — сгруппировать работу; у задач можно проставлять оценку (story points),
  по спринту считаются velocity/burndown.

Всё это доступно и агенту через инструменты `agile_*` (адъютант может вести доску по вашей просьбе:
«заведи спринт», «перекинь задачу в ревью», «оцени задачу в 3 балла»).

### Группы в Telegram (командная работа)
1. Добавьте бота в групповой чат. **Важно:** чтобы бот начал вести проект, хотя бы один участник
   должен быть уже зарегистрирован (сделать `/start` боту в личке) — иначе бот не отвечает и не
   создаёт проект (защита от случайного добавления). Администратор инстанса может разрешить конкретные
   чаты через `NABU_GROUP_ALLOWLIST`.
2. Первое сообщение (от известного пользователя) создаёт **пространство проекта** для этой группы.
3. Пишите в чат как обычно — адъютант различает участников по имени, память и задачи проекта общие
   для команды, но изолированы от личных пространств участников.

## 5. Программирование проектов (песочницы)

Адъютант может писать и запускать код в **изолированной docker-песочнице** проекта:

- Только папка проекта монтируется в контейнер; по умолчанию **без сети**, с лимитами
  CPU/памяти/процессов и жёстким таймаутом.
- Инструменты: записать/прочитать файл, выполнить команду (`network:true` для npm/pip/git), `git clone`,
  `git commit` (локально), `git status`.
- **`git push` — только с вашего подтверждения.** Когда агент захочет запушить наружу, он попросит
  одобрение (кнопка в чате/Telegram). Одобрение одноразовое и привязано к конкретному проекту; токен
  доступа к репозиторию не попадает в логи. Приватные репозитории — клонирование по URL с токеном,
  который вы предоставляете через то же одобрение.

Просто скажите адъютанту: «склонируй репозиторий X», «добавь функцию Y и закоммить», «запушь в main» —
последнее спросит подтверждение.

## 6. Spaces — публикация сайтов

Любой проект можно опубликовать как **статический сайт** (как GitHub Pages):

- Адъютант генерирует сайт из Markdown-файлов проекта и публикует его.
- Публичный адрес: `https://<ваш-домен>/s/<slug>/` — доступен всем по ссылке.
- Снять с публикации — space снова приватный.

Скажите: «опубликуй проект как сайт под адресом my-notes» → получите ссылку `/s/my-notes/`.
Slug уникален; занятый — попросит другой.

## 7. Маркетплейс агентов

Вкладка **Агенты** (в 🛠 OlimpOS):

- Список **общих** агентов сообщества (`__commons__`) с числом использований.
- **Опубликовать своего** агента: введите его slug → **Опубликовать**. После этого им смогут
  пользоваться другие. Личные агенты остаются приватными, пока вы явно их не опубликуете.

Новые агенты порождает `agent-creator` по реальной потребности (нехватка компетенции) — не заранее.

## 8. Приватность и безопасность — что гарантируется

- Личная память/знания **не** смешиваются между пользователями (жёсткий инвариант, скоуп по `user_id`
  и `namespace`, fail-closed).
- Эмбеддинги считаются **локально** (Ollama); `private`/`vault` не уходят во внешние AI/embedding-API и логи.
- Высокорисковые действия (`git push`, внешние вебхуки, деструктивные операции) — **только через ваше
  подтверждение**; модель не одобряет собственные действия.
- Сессии подписаны, cookie `HttpOnly`; приложение доступно только за HTTPS-прокси.
- Совет **советует** — финальное решение всегда за вами. Доменные агенты дают информацию и структуру,
  но не заменяют лицензированных специалистов (врача/юриста/финсоветника/терапевта).

## 9. Частые вопросы

**Мои данные видит другой пользователь?** — Нет. Личные пространства изолированы; общими становятся
только явно опубликованные агенты и обезличенный опыт.

**Что видит команда в групповом проекте?** — Общую память и задачи этого проекта. Личные пространства
участников остаются закрытыми.

**Могу ли я работать и лично, и в команде одним аккаунтом?** — Да. Личка с ботом → личное
пространство; групповой чат → пространство проекта. Аккаунт один.

**Кто может сделать `git push`/опубликовать сайт?** — Действие инициирует агент по вашей просьбе, но
`git push` требует явного подтверждения; публикация сайта делает space публичным по ссылке.

**Забыл пароль от email-аккаунта?** — Войдите через Telegram (тот же аккаунт, если Telegram уже
привязан), либо обратитесь к владельцу инстанса.
