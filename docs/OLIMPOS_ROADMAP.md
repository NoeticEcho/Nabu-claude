# OlimpOS — Roadmap: Nabu → много-тенантная коллаборативная платформа

> Статус: **черновик архитектуры** (2026-07-07). Превращение Nabu-claude из однопользовательского
> локального инструмента в много-тенантную платформу на VPS: команды/организации, проекты,
> программирование, spaces/сайты, agile, общий банк агентов. Кодовое имя модуля — **OlimpOS**.
>
> Это НЕ один проект, а платформа из ~7 подсистем. Каждая фаза → своя спека → план → реализация.
> Данный документ — верхнеуровневая карта и последовательность.

## 0. Что уже есть (фундамент, переиспользуем)

- **Двухосевой скоупинг** `namespace` (память/знания/личность/совещания/скиллы) + `user_id`
  (домен/задачи/финансы/здоровье) — вшит в КАЖДЫЙ репозиторий через `buildDeps({namespace,userId})`,
  fail-closed при половинчатом профиле (`lib/src/index.ts`). Таблицы `users`, `mem_namespace` есть.
- Доменная схема: `projects / tasks / goals / quests` (основа agile). `procedures` (процедурная память).
  Agent-memory, personality, effectiveness — namespace-scoped.
- Агенты — 74 `.md` в репо (общие). Функциональный конвейер, Совет, agent-creator, critic.
- Docker-стек (Postgres+pgvector, TypeDB, Ollama). Демон = web-чат(SSE) + Telegram-бот + планировщик.
- **Commons** (`docs/COMMONS.md`) — федеративное самоулучшение с приватностью-агрегатами: прообраз
  «общего банка агентов/скиллов/опыта».
- Приватность: `visibility` (default/private/vault), vault E2E без эмбеддингов, гейт remote-эмбеддинга.

## 1. Ключевые инварианты платформы (не нарушать)

1. **Изоляция тенантов**: личная память/знания/чаты пользователя (episodic/semantic/personal
   knowledge/vault/chat) НИКОГДА не пересекаются между пользователями. Скоуп по `namespace` (личное
   пространство пользователя).
2. **Общий слой (commons)**: определения агентов, скиллы, процедурная память, агрегированный ОПЫТ
   агентов (эффективность, «что сработало») — общие, в выделенном shared-namespace; но опыт хранится
   как ОБЕЗЛИЧЕННЫЕ агрегаты/паттерны, без сырого личного контента (как Commons).
3. **Явное согласие на публикацию**: перенос чего-либо из личного в общий банк/рынок/публичный space
   — только по явному действию владельца.
4. **Auth обязателен для интернет-экспозиции**: localhost-trust отменяется для публичного инстанса;
   каждый запрос аутентифицирован и резолвится в тенанта до `buildDeps`.
5. **Sandbox-изоляция**: выполнение кода проектов — в изолированных docker-песочницах, без доступа к
   секретам/данным других тенантов и к хосту.
6. **Harness-дисциплина сохраняется**: узкие типизированные tools, approval для высокорисковых
   действий (внешние коммиты/деплой/платежи — через кнопку человека), бюджеты циклов.

## 2. Подсистемы и последовательность фаз

### Фаза 1 — Мультитенантный фундамент (P1) ⭐ прерогатива
Резолвинг тенанта на каждый вход, регистрация, разделение shared/private.
- `users` (id, tg_user_id, email, pass_hash, personal_namespace, created_at) + `tenant`-резолвер:
  Telegram `from.id` → user → его `namespace/userId`; web-сессия → user.
- Регистрация: Telegram `/start` в личке бота → создать user + личный namespace + топики; web —
  email/пароль или «войти через Telegram».
- **SHARED-namespace** (`__commons__`) для агентов/скиллов/процедур/опыта; личный namespace на юзера.
- Демон обслуживает МНОГО чатов/пользователей (убрать единый `boundChatId` → маппинг chat→tenant).
- Спека: `docs/specs/2026-07-07-p1-multitenancy-design.md`.

### Фаза 2 — Веб-платформа (интернет) (P2)
- Реальный auth: email+пароль (argon2/scrypt) + «войти через Telegram» (deep-link + одноразовый код).
- Сессии (httpOnly cookie), CSRF (уже есть), rate-limit, отмена localhost-trust для публичного режима
  (флаг `NABU_PUBLIC=1` включает auth-обязательность). Reverse-proxy/HTTPS-гайд.
- Мультиюзерный веб-чат: сессия → тенант → scoped deps. UI регистрации/логина.

### Фаза 3 — Групповая коллаборация (P3)
- Групповые Telegram-чаты: несколько tg-пользователей + Nabu над ОБЩИМ проектом. Адъютант различает
  участников (`from.id`), атрибутирует реплики, ведёт общий контекст проекта + личные грани.
- Группа ↔ проект(ы); топики адаптируются под задачи проекта. Членство/роли (owner/member/viewer).
- Общее пространство ПРОЕКТА (project-namespace) отдельно от личных пространств участников.

### Фаза 4 — Общий банк/рынок агентов (P4)
- Реестр агентов в БД (не только файлы): пользовательские агенты → общий банк, discoverable всеми.
- Общие скиллы + процедурная память + агрегированный опыт (effectiveness/recommendation across users)
  → агенты эффективнее для каждого. Приватность-агрегаты (без личного сырья).
- «Рынок»: рейтинг/использование агентов; форк/адаптация под свой проект.

### Фаза 5 — Программирование проектов (OlimpOS-dev) (P5)
- Docker-песочницы на проект (изолированный код-ран, лимиты, без секретов чужих тенантов).
- Написание кода/спек, git-интеграция (GitHub/GitLab: clone/branch/commit/PR — через approval).
- Роли софт-компании: уже есть creator-агенты (software-dev/web-dev/entrepreneurship/copywriting);
  добавить product/marketing/QA/devops-роли и оркестрацию «команды».

### Фаза 6 — Spaces + генерация сайтов (P6)
- Spaces на пользователя/проект: публичные (доступ по URL, как GitHub Pages) и приватные.
- Генерация сайтов: **Astro** (SSG, острова, MD/MDX-контент) как дефолт — из контента проекта/знаний.
- Сервинг: демон/reverse-proxy отдаёт публичные spaces по пути `/s/<space-slug>`; приватные — за auth.

### Фаза 7 — Agile / проектное управление (P7)
- Поверх `projects/tasks`: эпики, спринты/итерации, доски (kanban), оценка, назначение на участников.
- Персонально и командно. Метрики (velocity, burndown) через существующий analytics-слой.

## 3. Зависимости

```
P1 (мультитенант) ─┬─► P2 (web-auth) ──► P3 (группы) ──► P4 (банк агентов)
                   └─────────────────────► P5 (dev/sandbox) ──► P6 (spaces/сайты)
P3 ──► P7 (agile)     P5 ──► P7
```

P1 — прерогатива всего. P2 и P5 могут идти частично параллельно после P1. P4/P6/P7 — надстройки.

## 4. Риски и решения (для владельца)

- **Безопасность интернет-инстанса** — крупнейший сдвиг: localhost-trust → полноценный auth + изоляция
  песочниц + аудит. Делаем поэтапно, публичный режим за флагом, дефолт остаётся приватным/локальным.
- **Схема — только аддитивно** (инвариант). Новые таблицы (memberships, spaces, sandboxes, agent_registry,
  epics/sprints) добавляются миграциями, существующие не ломаем.
- **Ресурсы VPS**: песочницы и SSG требуют CPU/RAM; лимиты на тенанта/проект, очереди.
- **Приватность общего опыта**: агрегаты агентов не должны утаскивать личный контент — тот же принцип,
  что в Commons (только паттерны/метрики).

## 5. Статус реализации

- ✅ **P1 (мультитенант-фундамент)** — ГОТОВ и верифицирован. Миграция 019 (users укреплён,
  membership, agent_registry, `__commons__`), `lib/tenancy.ts` (резолвер + web-регистрация scrypt),
  Telegram-wiring (per-tenant cid/session/scope), extraEnv→MCP. Изоляция памяти проверена вживую.
- ✅ **P2 (веб-auth)** — ГОТОВ. `cli/web-auth.mjs` (сессии/cookie, register/login/logout/me),
  per-request tenant-скоуп из сессии, login-UI в SPA. Всё за флагом `NABU_MULTITENANT=1`; дефолтный
  локальный режим (localhost-trust) не изменён. Auth-флоу проверен на живом сервере.
- ⏳ **P3–P7** — в бэклоге (группы, рынок агентов, sandbox+git, spaces/сайты, agile). Спеки по подходу.

**Перед P3+ рекомендуется:** развернуть инстанс с `NABU_MULTITENANT=1` на VPS и прогнать реальный
мульти-юзер тест (2 Telegram-аккаунта + 2 web-логина) — подтвердить изоляцию в бою до надстроек.
Также решить по «login через Telegram» (deep-link) и HTTPS/reverse-proxy для публичного доступа.
