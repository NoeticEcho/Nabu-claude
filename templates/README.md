# Шаблоны document-synthesizer (агент #22)

Здесь лежат редактируемые markdown-шаблоны, по которым субагент `document-synthesizer`
собирает вторичные документы из памяти пользователя. Шаблоны загружает инструмент
`template-loader` по `template_id`.

## Вход агента
`{ template_id, subject_id, parameters }` — какой шаблон, вокруг какой сущности собирать,
и опциональные параметры (например, версия, язык). Выход — `.md` с заполненными разделами
и обязательными ссылками на исходные заметки.

## Устройство шаблона
- **YAML-frontmatter**: `template_id` (совпадает с именем файла без `.md`), `title`, `description`.
- **Тело**: разделы с плейсхолдерами и HTML-комментариями-подсказками, откуда брать данные.

## Соглашение о плейсхолдерах
- Формат — `{{placeholder}}`: строчные буквы, слова через `_` (например, `{{success_metrics}}`).
- `{{subject_id}}` и `{{date}}` — служебные, подставляются автоматически.
- Комментарии `<!-- откуда: … -->` — инструкции для агента, в готовый документ не переносятся.
- Раздел `{{source_notes}}` есть в каждом шаблоне и заполняется всегда.

## Что подставляет document-synthesizer
- Контекст: `nabu-memory.recall`, `nabu-memory.list_recent_episodes`, `nabu-pipeline.search_knowledge`.
- Связи вокруг subject: `nabu-memory.graph_neighbors` / TypeDB MCP.
- Таблицы (goals, quests, metric_*): Supabase MCP. Тяжёлую агрегацию — через `nabu-analytics`, не через Claude.

## Правило «только из памяти»
- Заполнять поля **только** данными из реальной памяти пользователя; ничего не выдумывать.
- Нет данных → писать «нет данных» или выносить в «Открытые вопросы», а не додумывать.
- Каждый документ ссылается на исходные заметки (`{{source_notes}}`); проходит через critic #03.
- Private/vault обрабатываются локально (Ollama), документ наследует приватность, в облако/логи не уходит.

## Файлы
- `project-passport.md` — паспорт проекта.
- `spec.md` — техническая спецификация.
- `literature-review.md` — литературный обзор.
- `manifesto.md` — манифест.
- `value-map.md` — ценностная карта.
