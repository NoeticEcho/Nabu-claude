# Security Policy

Nabu handles deeply personal data — health, finance, relationships, private
notes. Security and privacy are core design goals, not afterthoughts. We take
reports seriously.

## Supported versions

Nabu ships from a single moving line. Only the latest `master` is supported;
please reproduce issues against the current tip before reporting.

| Version | Supported |
|---|---|
| latest `master` | ✅ |
| older commits / tags | ❌ |

## Reporting a vulnerability

**Please do not open a public issue for security vulnerabilities.**

Report privately via GitHub Security Advisories:

> https://github.com/noeticecho/nabu-claude/security/advisories/new

Include, where possible:

- affected component (CLI, chat server, a specific MCP server, a guard hook, the
  vault, the Telegram bot, a schema file)
- reproduction steps and impact
- the commit you tested against and relevant `nabu doctor` output
- any suggested remediation

We aim to acknowledge reports promptly and will keep you updated on remediation.
Please give us reasonable time to fix an issue before any public disclosure.

## Security model

Nabu is designed to run locally and keep sensitive data off third-party
services. Key properties:

- **Localhost-only surfaces.** The web chat and daemon bind to `127.0.0.1` only,
  and requests are validated with a Host-header check. There is no exposed
  network listener by default.
- **Narrow, allowlisted tools in headless sessions.** Scheduled and chat-driven
  Claude Code sessions run with a tight tool allowlist (Nabu MCP tools + read +
  Task); routine file-write operations are not available to them.
- **Guard hook for destructive commands.** A `PreToolUse` guard hook blocks
  dangerous shell operations (covered by 47 test cases in `npm run test:hooks`).
- **Local embeddings.** Embeddings are computed locally via Ollama
  (`nomic-embed-text-v2-moe`); there is no cloud embedding path.
- **Vault encryption at rest.** `visibility: vault` content is encrypted with
  AES-256-GCM before it is written to Postgres, using a machine-local key
  (`NABU_VAULT_KEY`, generated by `nabu init` and kept only in your `.env`).
  Vault entries get no embeddings, and their plaintext never enters model
  context through routine paths — extraction runs on a local LLM.
- **Out-of-model approvals.** High-risk actions require explicit human approval
  via a button in the web chat or Telegram (`user:web` / `user:telegram` in the
  audit trail). The model cannot approve its own high-risk actions.
- **No message text in logs.** Chat exchange logs record metadata
  (`{ts, evt, threadId, ms, costUsd, ok}`) but never message bodies.

## Out of scope

- **Telegram transport privacy.** The optional Telegram bot routes message
  content through Telegram's servers. Enabling the bot is a deliberate channel
  choice by the user; do not send `vault` content through it. This is documented
  behavior, not a vulnerability.
- **Loss of `NABU_VAULT_KEY`.** Losing the key means vault records are
  unrecoverable by design. Back up your `.env`. This is expected behavior.
- **Compromise of the host machine.** Nabu's local-first model assumes the
  machine it runs on is trusted; local root access is outside our threat model.
- **User-supplied third-party API keys / external services** you configure
  beyond the documented local stack.
