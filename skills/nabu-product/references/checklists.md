# Чек-листы

## Перед новой фичей
- [ ] Есть подтверждённая проблема (job) из discovery, не из головы (Iron Law: discovery до delivery)
- [ ] Гипотеза: «[решение] улучшит [метрика] для [сегмент], потому что [причина]»
- [ ] Одна метрика успеха + порог
- [ ] **Kill-критерий записан ДО старта**: «свернуть, если к [срок] [метрика] < [порог]»
- [ ] Приоритет посчитан (RICE/ICE) с явными допущениями и честным Confidence
- [ ] Effort честный, с запасом; альтернатива «не строить» рассмотрена
- [ ] Записано: `nabu-improve.record_recommendation` / `nabu-domain.add_task`

## После релиза фичи
- [ ] Метрика снята против порога (а не «ощущается лучше»)
- [ ] Достаточна ли выборка для вывода? иначе — ждать, не решать на шуме
- [ ] Kill-критерий сработал? если да — сворачивать без торга с sunk cost
- [ ] Реальное использование vs ожидание (кто НЕ пользуется и почему)
- [ ] Исход — эпизодом в память (`remember_episode`): гипотеза → факт → урок

## Ежемесячная продуктовая ревизия
- [ ] Retention-когорты: кривая выполаживается или течёт в ноль?
- [ ] Активация: доля и время до «ага-момента» по свежим когортам
- [ ] Roadmap now/next/later актуализирован; WIP ограничен (не всё «сейчас»)
- [ ] Что удалить/упростить? (продукт растёт и вырезанием, не только добавлением)
- [ ] Внешние обещания дат сверены с реальностью (Iron Law: roadmap ≠ обещание)

## Discovery-интервью
- [ ] Цель — понять job и обстоятельства, не продать и не проверить свою идею
- [ ] Вопросы о прошлом поведении, не о гипотетическом будущем
- [ ] Не называю свою фичу; не спрашиваю «понравилось бы X?»
- [ ] Пауза и «расскажите подробнее» вместо подсказки
- [ ] Отделяю вежливость/комплимент от реального сигнала спроса
- [ ] Заметки — в память как `private` (данные пользователей чувствительны)
