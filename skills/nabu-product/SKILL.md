# Продуктовая стратегия для solo/малых команд (доменный пак)

Тонкий вход; глубина — в `references/`. Пак повторяет форму `nabu-marketing`. Помогает решить,
что строить и что НЕ строить, на основе job-to-be-done и данных — без обещаний, что фича «выстрелит».

## Iron Laws (незыблемо)

1. **Не выдумывать пользовательские данные и фидбек** — только реальное из памяти
   (`nabu-analytics`, интервью в памяти) или честное «данных нет, вот как собрать».
2. **Kill-критерий обязателен ДО старта фичи** — заранее оговорённый порог/срок, при котором
   фичу сворачивают. Нет kill-критерия → фичу не начинать.
3. **Discovery до delivery** — сначала подтвердить проблему и спрос, потом строить. Не наоборот.
4. **Roadmap — не обещание дат наружу** — now/next/later внутри; наружу даты только с явной
   оговоркой о неопределённости. Обещанная клиенту дата = обязательство, не план.
5. **Продуктовые метрики ≠ метрики тщеславия** — активация/retention/revenue, не скачанные/лайки/
   регистрации. Оценки RICE/ICE — это оценки, не факты (помечать допущения).

## Мини-цикл (по умолчанию)

1. Понять стадию: до product-market fit → discovery и валидация проблемы дёшево; после →
   углубление того, что удерживает. Не строить масштаб на неподтверждённой проблеме.
2. Одна проблема (job) → одна гипотеза решения → одна метрика успеха → один kill-критерий.
   Расширение — после сигнала, не «на всякий случай».
3. Каждая фича — как **ставка с условиями выхода**: гипотеза, метрика, срок, kill-критерий.
   Фиксировать через `nabu-domain.add_task` / `nabu-improve.record_recommendation`.

## Интеграция с Nabu

Позиции в Совет — через агента `product-strategy` (канон `agents/product-strategy.md`). Реальные
метрики — `nabu-analytics.list_metrics`/`aggregate_metric`; проекты/задачи/цели — `nabu-domain`.
Спрос/канал → агент `marketing`; запросы клиентов → `sales`; осуществимость → `software-dev`;
приватность сбора данных → наружу к юристу (`SAFETY.md`). Данные интервью чувствительны → `private`.
