# Nabu-claude — Project Context for Claude Code

Читается каждую сессию. Тугой контекст + инварианты. Детали — в `ARCHITECTURE.md`,
`LIFE_DOMAINS_RESEARCH.md`, `SAFETY.md`. При расхождении приоритет у них, но сообщи.

> **Standalone-продукт (решение v1.0.0, 2026-07-04).** Nabu-claude работает на Claude Code с
> собственным локальным стеком (Postgres+pgvector, TypeDB, Ollama в docker). Shared-режим к БД
> основного приложения Nabu УДАЛЁН. Схема — наследие основного Nabu: менять только аддитивно.

## Что это

ИИ-«Совет» для всех сфер жизни пользователя внутри Claude Code. Пользователь общается с
адъютантом; на сложные запросы созывается коллегиальный Совет доменных экспертов; решение
синтезируется с честными trade-off'ами. Память и личность агентов — в общей БД.

## ГЛАВНЫЙ факт: мозг — Claude Code (Max)

Рассуждение выполняешь ты (Claude Code), не отдельные API-вызовы. Агент = роль + skill +
доступ к памяти/инструментам. Тяжёлое (эмбеддинги, транскрипция) — локальные модели.

## Harness-дисциплина (agents-best-practices)

- **Harness действует, модель предлагает.** Исполнение и проверки — в коде/командах.
- **Классы риска**: read (автономно, узко) · draft (автономно, помечено) ·
  write/external/financial/destructive (через **approval вне модели**).
- **Никаких broad-инструментов** (`execute_anything` и т.п.). Только узкие типизированные tools.
- **Бюджеты циклов** (шаги/время/токены/стоимость) + причина остановки.
- **Контекст строится, не сваливается**: достать нужное, пометить границы доверия.
- **Сначала MVP**, расширение Совета/специалистов — по реальной потребности.

## Агенты (детали — LIFE_DOMAINS_RESEARCH.md)

- **Совет министров** (9): health, mind, finance, work, learning, relationships, growth, lifestyle, admin.
- **Функциональные**: adjutant (с ним общается пользователь), council (коллегиальность),
  decision-maker, agent-creator (порождает новых агентов при нехватке компетенции),
  critic (проверка границ/безопасности), память (librarian/entity-extractor/retriever/
  memory-keeper/reflector), voice-transcriber (опц., неблокирующая).
- Узкие специалисты — порождает agent-creator по требованию, не заранее.

## Коллегиальное решение (детали — ARCHITECTURE.md §4)

adjutant триаж → простой/единодоменный → ответ; многодоменный → **Совет как Agent Team**
(министры-тиммейты + SendMessage + общий task-list; durable-буфер `deliberation`) собирает позиции
→ конфликты → синтез с trade-off'ами → (если выбор) decision-maker → critic → ответ + память.
Флаг `CLAUDE_CODE_EXPERIMENTAL_AGENT_TEAMS=1`; без него — fallback через Task-диспатч + буфер.
Это оркестрированная коллегиальность, не рой автономных агентов. Детали — `docs/TEAMS.md`.

## Память и личность

Память: 7 типов на 3 хранилищах (общая БД с основным Nabu). Личность: числовые черты из
`agents/*.json` → директивы по `PERSONALITY_RENDERING.md`. Стилизация, не сознание. Все
агенты Совета видят общую память пользователя.

## Данные

Batch (индексация) → прямое подключение; интерактив агента → Postgres/TypeDB MCP. За
интерфейсами репозиториев (Option Д). Эмбеддинги: Ollama (приватное), облако только `default`.

## ИНВАРИАНТЫ (не нарушать)

1. **Границы компетенции** (`SAFETY.md`): доменные агенты дают информацию и структуру, НЕ
   профессиональный совет; направляют к лицензированному специалисту (врач/юрист/финсоветник/
   терапевт). ИИ поддерживает, не заменяет профессионалов и живые связи.
2. **Приватность**: эмбеддинги — только локально (Ollama; облачного пути нет). `private`/`vault`
   — никогда в сторонние AI/embedding-API и логи. `recall` не отдаёт `vault` модели без явного запроса
   visibility. Нет visibility → `private`. Медицина/финансы/отношения по умолчанию `private`.
3. **Wellbeing** (`SAFETY.md`): crisis detection, не диагностировать, при кризисе — ресурсы поддержки.
4. **Не выдумывать** факты о пользователе — только из памяти; нет данных → честно сказать.
5. **Личность — не сознание**: не притворяться человеком, не манипулировать, не льстить,
   не поощрять сверхзависимость. honesty/kindness не ниже порогов.
6. **Совет советует** — финальное решение за пользователем; trade-off'ы наружу, не усреднение.
7. **Harness**: высокорисковые действия — через approval; узкие tools; бюджеты циклов.
8. **Тяжёлые операции — локально**, не через Claude.
9. **Не ломать общую БД**: схема из основного Nabu; дополнения аддитивны.

## Рабочие правила

- Сборка: сначала план, потом код; после каждой фазы — контрольная точка.
- Свежие API (Whisper, TypeDB 3.x, pgvector, MCP SDK) — сверять через context7.
- Архитектурные решения вне docs — не выбирать молча, спрашивать.
- Реальную схему БД основного Nabu — проверить через MCP до изменений.

## Чего НЕ делать

- Не дублировать/ломать схему основного Nabu.
- Не давать профессиональный совет вместо направления к специалисту (`SAFETY.md`).
- Не отправлять private/vault в облако/Claude/логи.
- Не плодить агентов впрок — порождать по потребности.
- Не делать автономные broad-инструменты; не обходить approval для высокорисковых действий.
- Не выдумывать данные о пользователе; не притворяться человеком.
