# VERSIONING — политика версий Nabu

Nabu следует **[SemVer 2.0.0](https://semver.org)**: `MAJOR.MINOR.PATCH`.

## Почему это понадобилось

К версии 1.9.0 продукт пришёл слишком быстро: `MINOR` бампался почти на каждый набор фиксов, а
не только на новые фичи. Версия росла быстрее сущности. Мы **не сбрасываем** номер (1.x
опубликован на GitHub Releases — сброс к 0.x запутает), но с этого момента **дисциплинированы**:
считаем изменения по SemVer и релизим осмысленно, а не на каждый коммит.

## Публичный контракт Nabu (что считается «API»)

Ломающим (`MAJOR`) считается несовместимое изменение любого из:
- **CLI-команды** `nabu …` — удаление/переименование команды или её обязательных флагов.
- **MCP-tools** (8 серверов) — удаление/переименование tool или несовместимое изменение его входной схемы (агенты и скиллы зависят от имён/контрактов).
- **Формат конфигов** — `config/*.json`, `profiles.json`, `schedule.json`, `.env`-ключи (удаление/переименование).
- **Манифест плагина** `.claude-plugin/plugin.json` — структура.
- **HTTP-API демона** (`/api/*`) — контракт эндпоинтов.
- **Схема БД** — по инварианту она **только аддитивная**, поэтому ломающих схемных изменений быть не должно; если однажды понадобится не-аддитивное — это `MAJOR` + миграция.

## Правила бампа

| Уровень | Когда | Примеры |
|---|---|---|
| **MAJOR** (`X`.0.0) | Ломающее изменение публичного контракта (см. выше). | удалили команду, переименовали MCP-tool, сменили формат `.env` |
| **MINOR** (`x`.`Y`.0) | Новая обратно-совместимая функциональность. | новая команда/агент/MCP-tool, новая фича (напр. библиотека знаний), новый импорт |
| **PATCH** (`x`.`y`.`Z`) | Обратно-совместимые фиксы и внутреннее. | багфиксы, приватность-харднинг, рефакторинг, доки, перф |

Правила выбора: если есть хоть одно ломающее → MAJOR. Иначе если есть хоть одна новая фича → MINOR.
Иначе → PATCH. **Приватность/безопасность-фиксы — это PATCH** (если не добавляют фичу), но их всегда
отмечаем в CHANGELOG отдельной пометкой.

## Каденс

- Релизим **накопленными наборами**, а не покоммитно. Несколько фиксов и один рефакторинг → один
  PATCH-релиз; фича со своими фиксами → один MINOR.
- Каждый релиз: bump версии во всех файлах (скрипт ниже) → запись в `CHANGELOG.md` → git tag `vX.Y.Z`
  → GitHub Release (`gh release create`, не только тег — тег сам релизом не становится).

## Инструмент

`scripts/version.mjs` — единая точка бампа (раньше версия правилась вручную в ~8 файлах, дрейф):

```bash
node scripts/version.mjs            # показать текущую + предложить bump по коммитам с последнего тега
node scripts/version.mjs patch      # 1.9.0 → 1.9.1 во всех файлах
node scripts/version.mjs minor      # 1.9.0 → 1.10.0
node scripts/version.mjs major      # 1.9.0 → 2.0.0
node scripts/version.mjs auto       # bump по conventional-commits (feat!→major, feat→minor, иначе patch)
```

Соглашение коммитов (влияет на `auto`): `feat: …` (MINOR), `fix: …`/`docs:`/`refactor:`/`chore:`
(PATCH), `feat!: …` или тело с `BREAKING CHANGE:` (MAJOR).

## Файлы, где живёт версия

`package.json`, `lib/package.json`, `mcp/*/package.json`, `.claude-plugin/plugin.json`,
`.claude-plugin/marketplace.json`, `mcp/*/src/index.ts` (`new McpServer({version})`),
`agents/registry.json`. Скрипт обновляет их все.
